# Phase 06.3: Walkthrough & Review Stage Design Alignment - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Pixel-match the walkthrough step cards, inline comment threads, review findings UI, submission panel, and DiffViewer inline integration to the updated design.html mockup. This is primarily a visual refinement phase — no new MCP tools, no new server capabilities. The one functional addition is the Valid/Not Valid toggle on inline finding cards (with dismiss + collapse behavior). All other new interactive elements shown in the design (suggested changes, Mark resolved/Needs work/Block) are deferred.

</domain>

<decisions>
## Implementation Decisions

### Walkthrough step banner
- **D-01:** Implement a step info banner above the diff in Walkthrough mode showing: "STEP X OF Y · N/M reviewed", step title, step commentary/description, "✓ Reviewed" badge (when step is visited), and prev/next navigation cards ("PREV [step title]" / "NEXT [step title]"). Use a flat background — do NOT add the gradient shown in the design.
- **D-02:** Prev/next cards are clickable and fire `walkthrough.stepAdvanced` events (same as the existing step click handlers).

### Walkthrough step cards (right panel)
- **D-03:** Pixel-match walkthrough step cards to the design: green checkmark circles for completed steps, numbered circles for pending, file path in muted monospace text below description, subtle card borders and spacing.
- **D-04:** Active/current step card gets a "VIEWING" badge and an "Ask about this step" button. The Ask button sends a chat request through `postUserRequest` with context about the current step (step title, file path, commentary).
- **D-05:** Cards match the design's typography, spacing, border-radius, and shadow treatment pixel for pixel.

### Inline threads & comments
- **D-06:** Full restyle of `ThreadCard` to match the design: severity-colored header bar showing "[SEVERITY] Line N · [category]" with close button, proper Claude/You avatar circles, styled message bubbles with timestamps, and a reply input field with Reply button at the bottom.
- **D-07:** Move ThreadCard from inline styles to CSS classes for consistency with the rest of the design system.
- **D-08:** Add a "Reply..." text input with Reply button at the bottom of each open (unresolved) thread card. Submitting a reply sends through the existing `postUserRequest` inline_comment path with the thread's lineId.
- **D-09:** Restyle `InlineComposer` to visually align with ThreadCard treatment — same border style, input styling, button styling. Consistent visual language for all inline comment surfaces.
- **D-10:** Defer: suggested change UI with Apply/Dismiss (DIFF-V2-03), Mark resolved/Needs work/Block action buttons. These are new features, not visual alignment.

### Findings cards (Review step right panel)
- **D-11:** Pixel-match findings cards to the design: active card left border + background highlight, dual badge layout (severity + category side by side, e.g., "BLOCKER SECURITY"), sparkle icon on Ask button, file icon + path:line reference.
- **D-12:** Clicking a finding card scrolls the diff to the finding's line AND opens/expands the ThreadCard at that location (if a thread exists from the self-review). Active card in the right panel highlights to show which finding is expanded.

### Finding validity toggle
- **D-13:** Add "IS THIS FINDING VALID? ✓ Valid × Invalid" toggle on each inline finding card in the diff (above the thread content). This is a user decision surface for each self-review finding.
- **D-14:** Marking a finding as Invalid: collapses the inline finding card in the diff, dims the card in the right-panel findings list (reduced opacity/strikethrough), updates finding counts (e.g., "5 open" → "4 open, 1 dismissed"). The finding persists in state but is deprioritized.
- **D-15:** A new `SessionEvent` variant (`finding.validitySet`) tracks the validity state change: `{ findingId, validity: 'valid' | 'invalid' }`. Follows the Phase 2 reducer/applyEvent/SSE pattern.
- **D-16:** `ResolvedFinding` type gains an optional `validity?: 'valid' | 'invalid'` field. The reducer updates the finding in `selfReview.findings` when `finding.validitySet` fires.

### DiffViewer inline integration
- **D-17:** Colored numbered finding markers in the diff gutter: red circles for blocker, amber for warning/major, pale amber for minor/nit. Numbered to match the finding card index in the right panel. Clicking a marker scrolls the right-panel findings list to that card.
- **D-18:** Lines with findings get a subtle severity-colored background tint: faint red for blocker lines, faint amber for warning/major, faint yellow for minor/nit. Tint must not overpower the green/red added/removed line colors.
- **D-19:** Inline thread cards expand as full-width push-down panels within the diff area with the new design styling (severity header, avatar messages, reply input).
- **D-20:** In Walkthrough mode, the currently-active step's hunk gets a subtle left border or background highlight to show which hunk the user is reviewing.
- **D-21:** File header bar above each file's diff pixel-matched to the design: path with directory/file split styling, +/- stat badges, Unified/Split toggle buttons, ✓ Mark reviewed checkbox.

### Submission panel (right panel)
- **D-22:** Restyle `SubmissionPanel` to match the design: recap stats table (Stages completed, Findings count with severity badges, CI Checks status, Walkthrough progress), Claude verdict suggestion card with "Why?" link and reasoning text, "Open submission dialog" CTA button. All data already exists in session state.

### Claude's Discretion
- Exact colors for severity tints on diff lines (must complement existing green/red line colors)
- Exact border-radius, shadow values, and spacing to match design.html
- How the "Why?" link on the Claude verdict suggestion card works (could open a chat message or expand inline)
- Exact animation/transition for finding dismiss + collapse
- Whether the file header directory/file split uses a `/` separator or breadcrumb styling
- Avatar circle colors and sizing for Claude vs You indicators

</decisions>

<specifics>
## Specific Ideas

- The design.html at project root is the single source of truth for all visual decisions. Serve locally (`npx serve .`) and navigate to `/design.html`, then click through Summary → Walkthrough → Review → Submission steps to see each view.
- Walkthrough step banner should feel lightweight — it provides context without stealing vertical space from the diff. Flat background, not the gradient from the design.
- The "Ask about this step" button on walkthrough cards and the "Ask" sparkle button on finding cards both route through the existing `postUserRequest` chat mechanism — no new server infrastructure needed.
- Finding validity (Valid/Invalid) is the one piece of new functionality in this phase. It needs a new SessionEvent variant and reducer branch, but follows the exact same pattern as every other event in Phase 2's pipeline.
- Thread reply input on ThreadCard reuses the existing `postUserRequest` inline_comment path — the infrastructure from Phase 06.1 handles all the routing.

</specifics>
