## 1. System Classification

**System Type:** Structured Data Extraction (inverted topology — the LLM is the tool _consumer_ producing schema-conformant output; the plugin is the tool _producer_ enforcing structure via zod inputSchemas)

**Description:**
Phase 4 introduces the first LLM-driven product surface of the git-review plugin. The plugin exposes two new MCP tools — `set_pr_summary` and `run_self_review` — that the user's active Claude Code session invokes. `set_pr_summary` accepts a structured summary (intent classification, key changes, risk areas, paraphrase of the PR description) and replaces the review session's summary atomically. `run_self_review` accepts an adversarial self-review payload (findings grouped by checklist category, each severity-tagged `blocker|major|minor|nit`, each anchored to a `lineId` the server resolves to `file:line` at handler time, with a verdict that defaults to `request_changes`). Both tools' zod schemas are the structural contract; both tools' descriptions carry the full prompting discipline the Claude Code session reads. The plugin itself makes zero LLM API calls. "Good" looks like: the Claude Code session produces a paraphrase-faithful summary, classifies intent correctly, and argues a `request_changes` verdict down to `approve` (rather than up from `approve`) when it finds genuine issues, with nit discipline enforced structurally and file:line anchors resolving against real diff lines.

**Critical Failure Modes:**
1. **Sycophantic self-review defaulting to approve** (ROADMAP success criterion 5, Pitfall 4) — on a fixture PR with a genuine bug the first pass produces `verdict: "approve"` with no or only nit findings, proving the adversarial framing + default-verdict inversion did not bite.
2. **File:line anchor hallucination** — `run_self_review` findings reference `lineId`s that don't resolve, or (pre-resolution) cite `file:line` positions that don't exist in the diff. The lineId-only rail (D-04) is a structural mitigation; evals must still verify the Claude Code session uses it correctly rather than inventing paths.
3. **PR-description ignored / paraphrase drift** (Pitfall 11) — summary intent doesn't match the PR description's stated intent, or the paraphrase field re-summarizes the diff instead of paraphrasing the author's words. Makes D-21 intent → review-lens coupling cascade wrong.
4. **Nit flood / severity inflation** (Pitfall 3) — more than 3 nits in a self-review payload (schema rejects; verify rejection path works AND the corrective message actually produces a better retry), or genuine issues downgraded to `minor`/`nit` to inflate the apparent verdict toward `approve`.
5. **Schema conformance failure on retry** — when zod rejects (nit cap breach, severity enum breach, verdict enum breach, malformed `lineId`), the Claude Code session must retry with a corrected payload. A non-actionable error message or an error the session can't recover from silently breaks the whole review loop.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling — AI-assisted code review for GitHub pull requests (and local branch diffs) driven by an MCP-connected Claude Code session.

**User Population:** A single senior software engineer (the author of this plugin) reviewing their own and other people's PRs. The user currently does LLM-assisted reviews by copy-pasting diffs into the Claude desktop app and finds that UX inadequate for real review work. There is no team, no multi-tenant deployment, no external customers — "domain expert" and "user" and "rubric calibrator" are the same person.

**Stakes Level:** Medium (bounded by human-in-the-loop).

The consequence ceiling is hard-capped by design: the user always reads the self-review and the generated summary *before* a single character goes to GitHub (Phase 6 submission is out of Phase-4 scope). But the *effective* stakes for Phase-4 output quality are higher than that ceiling implies, for two reasons:
1. **The Core Value promise — "produces a posted GitHub review of real reviewer quality"** — fails the moment the self-review is misleading enough that the user stops trusting it. On a solo tool, trust erosion is the primary failure mode; there is no PM or CI to rebuild it.
2. **Phase 4's structured output cascades into Phases 5 and 6.** `PrSummary.intent` is the input to Phase 5's walkthrough-ordering heuristic (D-21 intent → review-lens coupling). `SelfReview.findings` with their `blocker|major|minor|nit` severities feed Phase 6's pre-submit signal-ratio check (SUB-02). Bad intent classification or severity inflation at Phase 4 corrupts every downstream phase — a silent failure mode where the surface artifact (a self-review blob) looks plausible but the downstream computation it drives is wrong.

**Output Consequence:** The Claude Code session sees the summary + self-review, then uses them to drive hunk prioritization and inline-comment drafting in Phase 5 and a final GitHub review in Phase 6. The user reviews the self-review output in the left-hand findings sidebar and either regenerates it, proceeds to walkthrough, or ignores the low-severity findings. Nothing is auto-sent anywhere in Phase 4 — every posted review is user-gated by design.

### What Domain Experts Evaluate Against

Five rubric ingredients grounded in the senior-reviewer literature (Google eng-practices, Sadowski et al. 2018, CodeRabbit/Graphite practitioner discourse 2025-26) and the signal-vs-noise taxonomy from Jet Xu's low-noise code review framework. Each maps cleanly to one of Phase 4's five ROADMAP success criteria so the eval-planner can translate it into a measurable dimension in Section 5.

```
Dimension: Signal ratio (useful findings over total findings)
Good: At least 60% of findings are Tier-1 (observable failures: runtime bugs, security holes, broken API contracts, test-coverage gaps on changed behavior) or Tier-2 (architectural-consistency violations, measurable performance regressions, maintainability risks with concrete justification). Nits are at most 3 in the payload (schema-enforced via D-03) and each one is clearly marked and non-blocking.
Bad: More than 40% of findings are aesthetic / subjective / micro-optimization style suggestions with no measurable impact ("consider extracting", "this name could be clearer", "maybe memoize this"). Findings dump every style preference the reviewer has a hot take on. The review reads like a linter trying to justify its runtime.
Stakes: High
Source: Jet Xu, "Drowning in AI Code Review Noise" (2025) — explicit ≥60% signal ratio threshold for a "good" tool, ≥80% for "great"; practitioner discourse 2025-26 ("AI Is Quietly Destroying Code Review"); project Pitfall 3 (nit flood drowns critical findings) and ROADMAP success criterion 3 (findings ordered by criticality).
```

```
Dimension: Anchor correctness (file:line refs point where they claim)
Good: Every finding's clickable `file:line` reference lands on a line that actually exists in the diff AND is the line the rationale describes. On a changed line, the reviewer is commenting on what actually changed; on a context line, the rationale makes clear the comment targets pre-existing code (this is legitimate in Phase 4 — the `pre_existing: true` gate is Phase 5's problem, not Phase 4's).
Bad: Finding rationale discusses "the null check on line 42" but `file:line` points to an import statement, a blank line, or a non-existent line number; finding cites `src/foo.ts:120` when the diff only touches lines 80-95 in that file.
Stakes: Critical
Source: Project Pitfall 2 (LLM hallucinated line numbers — BLOCKER severity); diffray "LLM Hallucinations in AI Code Review" (2025); CodeRabbit hallucination incident reports showing how a single phantom security finding triggers real incident-response cycles. Structurally mitigated by D-04 lineId-only rail; eval still has to verify the session USES the rail correctly.
```

```
Dimension: Verdict calibration (the request-changes vs approve decision)
Good: On a PR with a genuine bug, the self-review lands at `request_changes` with at least one `blocker` or `major` finding that identifies the bug. On a clean PR with no issues, the self-review lands at `approve` or `comment` with zero `blocker`s and an honest accounting of what was reviewed. The reviewer argues DOWN from the default (request_changes → comment → approve), not UP from approve — verdict drift is visibly evidence-driven.
Bad: "LGTM" energy: a clean-looking `approve` or a thin `comment` on a PR with a real defect, with no findings in the category where the bug lives. Blocker findings downgraded to minor/nit to inflate the verdict toward approve. The self-review reads like social assent, not technical scrutiny.
Stakes: Critical
Source: Project Pitfall 4 (self-review becomes blandly positive); ROADMAP Phase-4 success criterion 5 (fixture PR with a genuine bug MUST NOT first-pass approve); Google eng-practices "Standard of Code Review" (reviewer must catch obvious red flags; nit-marking exists precisely to separate polish from must-fix); Sadowski et al. 2018 "Modern Code Review: A Case Study at Google" (defect-finding is the primary measured outcome).
```

```
Dimension: Coverage breadth (all five checklist categories seen, not just one)
Good: The `coverage` field carries a non-null status for every category (correctness, security, tests, performance, style). On a feature PR, `tests: partial` or `tests: fail` surfaces when tests are thin; on a refactor, `correctness: pass` carries the argument "behavior is preserved across these N call sites." Each status is justified by findings in that category — or by an explicit "pass" claim the reviewer is staking their credibility on.
Bad: Review fixates on one category (typically style or correctness) and leaves the others at `partial` or unset with no justification. Security is unevaluated on a PR that touches auth code. Tests are `pass` when there are no new tests for new behavior. Coverage chips in the UI go yellow across the board and the reviewer never interrogates their own blind spot.
Stakes: High
Source: Google eng-practices "What to look for in a code review" canonical category list (design / functionality / complexity / tests / naming / comments / style / documentation); SmithSpektrum code-review-interview assessment rubric; the built-in CHECKLIST const (D-02) makes these five categories the Phase-4 authoritative domain partition. ROADMAP success criterion 4 (per-category coverage visible in UI).
```

```
Dimension: PR-description fidelity (summary paraphrase and intent classification)
Good: The `paraphrase` field restates the PR description in the author's own framing, not a re-summary of the diff. On a PR titled "Fix null deref in auth middleware" the `intent` is `bug-fix` with high confidence; `keyChanges` enumerates the actual changes not some hallucinated larger feature; `riskAreas` calls out the security-adjacency of the touched code. When the author explicitly says "this is a refactor — no behavior change," that claim is preserved in the paraphrase and the self-review's review lens respects it (D-21 intent → review-lens coupling).
Bad: Paraphrase reads like an abstract of the patch ("this change modifies foo() and adds a test") rather than the author's stated goal. Intent is wrong (refactor PR classified as feature) or confidence is high despite ambiguous authorial signal. The self-review ignores the author's stated intent and reviews against the wrong lens (e.g., demands new test coverage on a pure refactor where behavior preservation is the claim).
Stakes: High
Source: Project Pitfall 11 (LLM ignores PR description); Bosu et al. Microsoft research on "Characteristics of Useful Code Reviews" (reviewer context-awareness is a primary useful-comment predictor); ROADMAP success criterion 1 (summary visibly paraphrases the PR description). D-06 structured intent+paraphrase fields; D-09 social-enforcement via UI visibility of the author-description comparator.
```

### Known Failure Modes in This Domain

Four domain-specific failure modes observed in production deployments of AI code reviewers (CodeRabbit, Graphite, Qodo, Cursor Review) as of 2025-26. These are distinct from the generic "hallucination" category — they're the specific shapes that hallucination and other LLM failures take in the code-review context:

1. **Phantom-security-finding cascade.** The LLM asserts a non-existent vulnerability (an SQL-injection risk in code that doesn't touch a database, a "missing CSRF check" on a route that has one) with high confidence and a convincing rationale. The reviewer forwards it to the team; a real incident response kicks off; an hour later someone traces back to a line that never existed. At published 29-45% false-positive rates on security-flavored findings (diffray 2025), this is not a tail risk — it's a weekly occurrence for teams using these tools. For this plugin, the equivalent damage is narrower (the user sees it before it posts, so no external blast radius) but the trust damage to the tool is the same: once the first phantom finding lands, the user stops treating subsequent findings as signal. Mitigation in Phase 4: adversarial framing demands evidence-per-finding (D-20); lineId-only anchoring (D-04) kills fabricated path:line strings at the schema boundary; the review is user-gated before any external surface sees it.

2. **Nit avalanche crowding out real findings.** The LLM, asked "review this PR," returns 15 comments: 2 are useful, 13 are "consider extracting this", "this variable name could be clearer", "maybe memoize this". Research on CodeRabbit specifically calls out that it's "the most talkative" tool and that high comment-volume correlates inversely with reviewer trust (Jet Xu 2025). Senior-reviewer discourse (Google eng-practices, dev.to "AI Is Quietly Destroying Code Review") treats nit-to-signal ratio as the single most predictive quality metric. Mitigation in Phase 4: schema-enforced nit cap of 3 (D-03) — not a prompt-level request, an ingress-time rejection with a corrective retry message.

3. **Intent mismatch with author narrative.** The LLM reviews against the wrong lens — demands new tests on a pure refactor, demands behavior-change justification on a bug fix, complains about code style on a hotfix. This is the Pitfall 11 failure and is well-documented in the AI-code-review comparison literature (Qodo 2026 "8 Best AI Code Review Tools" explicitly calls out "context-awareness to PR intent" as a differentiator). Mitigation in Phase 4: structured `intent` + `paraphrase` fields in `PrSummary` (D-06); tool description instructs the session to apply the intent-appropriate review lens (D-21); UI visibly pairs the paraphrase with the author's actual description for eyeball validation.

4. **Sycophancy drift over the course of a review.** On a long PR, the LLM's findings get softer as it accumulates reviewed hunks — it starts with critical scrutiny, drifts to "this looks fine", and by hunk 20 everything is `nit` or `minor`. Practitioners call this "review exhaustion" even though the LLM isn't actually tired; it's an artifact of context accumulation and the model's preference for agreement-shaped tokens. Mitigation in Phase 4: atomic `run_self_review` (D-01 — one shot, no drift-over-time) + default-verdict inversion (D-05 — the session has to argue DOWN from request_changes, not UP from approve) + criticality-ranked checklist that demands coverage across all five categories (prevents "I'll just run through style nits and call it done").

This is explicitly a personal developer tool (CLAUDE.md, PROJECT.md: "Audience: Single user (author)"), not a regulated production system. No PII, no PHI, no financial data, no GDPR subject rights, no SOC 2 audit surface. The only data flowing through Phase 4 is:

### Domain Expert Roles for Evaluation

For a solo-dev tool, there is no team of domain experts; the user/author IS the domain expert. Writing separate "Senior Practitioner" and "QA Engineer" roles here would be theater. Section 1b surfaces the honest shape.

| Role | Responsibility in Eval |
|------|------------------------|
| User-as-domain-expert (single senior engineer, the plugin author) | Reference dataset labeling (which fixture PRs exercise which failure modes); rubric calibration (what does *this* user consider a blocker vs. a nit on *this* user's typical PR shape); edge-case review (flagging cases where a published-practitioner rule diverges from the user's personal review style); production sampling (the user reviews every self-review output in normal daily use, so eval signal accumulates as a side-effect of using the tool). |
| (Implicit, not a person) Published senior-reviewer consensus | Second-opinion rubric anchor via the canonical sources: Google eng-practices, Sadowski et al. 2018, Bosu et al. Microsoft research, Jet Xu signal-vs-noise framework. When the user's personal rubric would diverge from the published consensus (e.g., user considers a certain nit legitimately blocking), Section 5 dimensions should flag the divergence rather than silently baking it in. |

**Known limitation to carry into Section 5:** the eval dataset is labeled by a single person who is also the tool's user. Sample-size of one, strong confirmation-bias risk (the labeler has priors on what the tool SHOULD flag). The eval-planner needs to design explicit mitigations:
- Fixture PRs should include "blind" cases where the user hasn't looked at the diff yet, labeled only against the published rubric sources, not the user's gut call.
- LLM-as-judge should be considered for at least the signal-ratio and anchor-correctness dimensions — both are mechanically checkable and don't require the domain-expert's taste.
- Regression anchoring: once a self-review passes on a fixture, any future regression on that fixture is a hard failure regardless of user's in-the-moment taste.

## 2. Framework Decision

**Selected Framework:** MCP TypeScript SDK (`@modelcontextprotocol/sdk`) + `zod` structured-output-via-tool-schemas

**Version:** `@modelcontextprotocol/sdk@1.29.0`, `zod@4.3.6` (both already installed and in production use in `server/package.json`)

**Rationale:**
The plugin architecture locks this entirely — it is not a selection from competing options. PROJECT.md states "the plugin does not make its own LLM API calls," and CONTEXT.md D-07 restates and operationalizes this: the user's Claude Code session is the sole LLM driver, invoking MCP tools whose zod inputSchemas ARE the structured-output contract. Every prompting surface — adversarial framing (D-20), paraphrase discipline (D-20), intent → review-lens coupling (D-21), default-verdict inversion (D-05), nit cap (D-03), severity enum (D-01), lineId-only anchor rail (D-04) — either lives in a static tool description the Claude Code session reads directly or is enforced by the zod schema at tool-invocation time. The stack is already installed and every new Phase-4 tool uses the same `McpServer.registerTool(name, { title, description, inputSchema }, handler)` pattern as Phase 3's `start-review.ts`. Adopting any additional framework would fight the locked architecture.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Claude Agent SDK | Presumes the framework user calls an LLM; here the LLM is the tool consumer, not the producer — wrong topology. |
| OpenAI Agents SDK | Same topological mismatch; also violates the Anthropic model-provider assumption of the Claude Code session. |
| LangChain structured output | Adds a dependency whose value (output parsers, retry loops, prompt templating) is redundant to zod validation at the tool boundary and to the Claude Code session's native retry semantics. |
| Vercel AI SDK | Aimed at applications that call LLMs; provides nothing useful for an MCP server whose LLM role is tool-consumer. |
| Custom prompt-template layer / "reviewer" prompt file | Rejected explicitly in D-20 in favor of inline tool-description strings the Claude Code session reads directly — no runtime templating, no indirection. |

**Vendor Lock-In Accepted:** No — the plugin is model-provider-agnostic by construction (it doesn't call any LLM). Lock-in is to the MCP protocol (open standard) and the `zod` library (ubiquitous TS validator). Both are explicitly acceptable per the project's tech stack in CLAUDE.md.

### Entry Point Pattern

The canonical shape for every Phase-4 tool. Copy-pastable against the existing `server/src/mcp/tools/start-review.ts` reference. Input schemas go through `Input.shape` because `registerTool` expects a **zod raw shape** (map of fieldName → ZodType), not a wrapped `z.object(...)`.

```ts
// server/src/mcp/tools/run-self-review.ts — shape sketch
import { z } from 'zod';
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { SessionManager } from '../../session/manager.js';
import { logger } from '../../logger.js';
import { CHECKLIST } from '../../checklist/index.js';

// Canonical severity / category / verdict enums — load-bearing for every findings schema.
const Severity = z.enum(['blocker', 'major', 'minor', 'nit']);
const Category = z.enum(['correctness', 'security', 'tests', 'performance', 'style']);
const Verdict = z.enum(['request_changes', 'comment', 'approve']);
const CoverageStatus = z.enum(['pass', 'partial', 'fail']);

// Finding shape: lineId-only anchors (D-04). Never accept freeform path/line.
const Finding = z.object({
  category: Category,
  checklistItemId: z.string().min(1),
  severity: Severity,
  lineId: z.string().regex(/^[A-Za-z0-9_-]+:h\d+:l\d+$/),
  title: z.string().min(1).max(200),
  rationale: z.string().min(1).max(2_000),
});

// Top-level input schema. D-03 nit cap via .refine on the findings array.
// D-05 default verdict via .default('request_changes').
const Input = z.object({
  findings: z
    .array(Finding)
    .refine((fs) => fs.filter((f) => f.severity === 'nit').length <= 3, {
      error:
        'Too many nits (>3). Promote the most important issue to `minor` or higher severity, and drop the rest. The nit cap is structural — see tool description.',
    }),
  coverage: z.object({
    correctness: CoverageStatus,
    security: CoverageStatus,
    tests: CoverageStatus,
    performance: CoverageStatus,
    style: CoverageStatus,
  }),
  verdict: Verdict.default('request_changes'),   // D-05: default-verdict inversion
  prKey: z.string().min(1),
});

// The full description string is what the LLM reads — where D-20 adversarial framing lives.
// Built from the TS-const checklist at server-start time, not at call time.
const DESCRIPTION = buildRunSelfReviewDescription(CHECKLIST);

export function registerRunSelfReview(mcp: McpServer, manager: SessionManager): void {
  mcp.registerTool(
    'run_self_review',
    { title: 'Run Self-Review', description: DESCRIPTION, inputSchema: Input.shape },
    async ({ findings, coverage, verdict, prKey }) => {
      try {
        // Server-side resolve lineId → (path, line, side) BEFORE applying the event.
        // Unresolvable lineIds → corrective isError:true so the Claude Code session can retry.
        const resolved = resolveFindings(manager, prKey, findings);
        const selfReview: SelfReview = {
          findings: resolved,
          coverage,
          verdict,
          generatedAt: new Date().toISOString(),
        };
        await manager.applyEvent(prKey, { type: 'selfReview.set', selfReview });
        return {
          content: [{ type: 'text' as const, text: renderSelfReviewAck(selfReview) }],
        };
      } catch (err) {
        logger.error('run_self_review failed', err);
        return {
          content: [{ type: 'text' as const, text: renderCorrectiveError(err) }],
          isError: true,   // LLM sees this, can self-correct; NOT thrown (would be protocol error)
        };
      }
    }
  );
}
```

### Key Abstractions

| Concept | What It Is | When You Use It (Phase 4) |
|---------|-----------|---------------------------|
| `McpServer.registerTool(name, config, handler)` | High-level SDK call that wires a named tool into the stdio JSON-RPC loop with zod validation. | Every new Phase-4 tool (`list_files`, `get_hunk`, `set_pr_summary`, `run_self_review`). |
| `inputSchema: Input.shape` (raw zod shape, not `z.object`) | The SDK consumes a zod **raw shape** — the object literal map of field → schema that `z.object()` wraps. `Input.shape` extracts it. | Every tool. Matches the `start-review.ts` convention already in-tree. |
| `tool `description` string` | Static text handed to the LLM as-is via `tools/list`. This IS the prompt. | The only place adversarial framing (D-20), paraphrase discipline (D-20), and intent → review-lens coupling (D-21) live. Long is fine. |
| `CallToolResult` (`{ content, isError? }`) return | The handler's return value. Setting `isError: true` signals an **LLM-visible** error — the session sees the `content` text and can retry. | Every recoverable failure path: malformed `lineId`, unknown `hunkId`, zod refinement failure surfaced as a corrective message, `prKey` not found. Never `throw` for recoverable errors. |
| Zod `.default(...)` + `.refine(...)` on tool input | Structural enforcement. `.default` changes missing input to a value (D-05 verdict default); `.refine` rejects invalid compositions (D-03 nit cap). | `run_self_review` verdict default + nit-cap refinement; `set_pr_summary` intent-confidence range; lineId regex gate. |

### Common Pitfalls

1. **Passing `z.object(...)` directly to `inputSchema` instead of `Input.shape`.** The SDK's `registerTool` expects a raw zod shape map (field → schema), not a wrapped `ZodObject`. Passing the wrapper compiles but validation at runtime is shaped differently from what the LLM sees via `tools/list`. Match `start-review.ts`: define `const Input = z.object({...})` then pass `inputSchema: Input.shape`.

2. **Throwing instead of returning `{ isError: true }` for recoverable errors.** Thrown exceptions bubble up as *protocol-level* JSON-RPC errors the LLM never sees — the Claude Code session just gets a server fault. Anything the session should retry (nit cap breach, unknown `lineId`, verdict enum mismatch) must return a `{ content: [{type:'text', text: correctiveMessage}], isError: true }` — the LLM reads the `text` and self-corrects. Thrown exceptions ARE converted by the SDK to `isError: true` automatically, but you lose control over the corrective message text. Explicit try/catch wins.

3. **`console.log` anywhere in the MCP server path corrupts stdio.** The MCP transport is JSON-RPC over stdout; any stray `console.log` injects non-JSON into the channel and the session hangs or dies cryptically. Always use `logger.error(...)` (stderr-only) per Phase 1 AP2. This has bitten other MCP implementers; don't re-learn it.

4. **Zod v4 `z.discriminatedUnion` disallows duplicate discriminator values across branches.** The `start-review.ts` `Source` schema documents this — the two `kind: 'github'` variants had to be wrapped in an inner `z.union` because v4 rejected the duplicate discriminator. Phase 4 schemas as sketched don't currently discriminate on a common key, but if the planner introduces one (e.g., a discriminated `Finding` with a severity-keyed payload shape), it will hit this. Use nested `z.union` or make the discriminator values unique.

5. **Tool response > ~2k tokens saturates the Claude Code session context.** Pitfall 5 is a BLOCKER for real PRs. Every Phase-4 tool response must stay in budget: `list_files` pages at 30 files (D-15), `get_hunk` slices within a hunk via cursor (D-16). `run_self_review` and `set_pr_summary` return a brief ack string only — their *effect* is the `SessionEvent` they emit through `applyEvent`, which fan-outs over SSE to the browser. Don't echo the full parsed payload back.

6. **MCP tool count cap is soft but real (~10 tools).** Pitfall 15. The tool-belt budget ends Phase 4 at 5/10 (D-14); planners must re-verify at each phase boundary. Tool *description length* is **not** capped — a 400-word `run_self_review` description is fine and is where adversarial framing has to live. What's capped is how many distinct tools the Claude Code session can usefully juggle before tool selection degrades.

7. **Silent replace on regenerate requires the reducer branch to replace unconditionally.** Both `summary.set` and `selfReview.set` overwrite `session.summary` / `session.selfReview` atomically (D-07, D-08). Don't partial-merge, don't compare generations, don't guard on `generatedAt`. The Phase-2 per-`prKey` Promise-chain queue already serializes the mutations, so there's no race to guard against — `applyEvent` is the serialization point.

## 4. Implementation Guidance

**Model Configuration:**
Not applicable in the usual sense — the plugin makes **zero LLM API calls**. The consuming LLM is the user's active Claude Code session; its model is whatever the user has configured in their Claude Code client (Claude Sonnet 4.6 / Opus 4.7 / etc. as of 2026-04). No model name, no temperature, no `max_tokens`, no API key live anywhere in this plugin's codebase.

The only "model configuration" surface Phase 4 owns is **implicit**: the *shape and wording* of each tool's `description` field controls how effectively the Claude Code session's model produces a conformant response. Treat the description strings as the configuration:
- **Description length:** long is fine (400 words for `run_self_review`), count is not (stay in Pitfall 15's ~10-tool budget).
- **Description structure:** role framing first ("your job is to find reasons to request changes"), then the checklist categories inline, then the schema expectations ("nit cap = 3", "default verdict = request_changes"), then the devil's-advocate stance prompts.
- **Implicit token budget:** the Claude Code session's own context window is what matters; Phase 4 tools collectively target a per-response ~2k-token ceiling (D-15 / D-16 pagination) so the session can make many tool calls without pushing other context out.

**Core Pattern:**

`run_self_review` is the load-bearing Phase-4 tool. It illustrates every concern that matters: adversarial-framing description, enum-locked severity, nit-cap refinement, lineId-only anchors, default verdict inversion, server-side resolution, atomic reducer mutation, LLM-visible corrective errors. `set_pr_summary` is the same shape with a simpler schema.

```ts
// server/src/mcp/tools/run-self-review.ts
import { z } from 'zod';
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { SessionManager } from '../../session/manager.js';
import { logger } from '../../logger.js';
import { CHECKLIST } from '../../checklist/index.js';
import type { SelfReview, ResolvedFinding } from '@shared/types';

// --- Enums are the structural contract --- //
const Severity = z.enum(['blocker', 'major', 'minor', 'nit']);              // D-01
const Category = z.enum(['correctness', 'security', 'tests', 'performance', 'style']);
const Verdict = z.enum(['request_changes', 'comment', 'approve']);
const CoverageStatus = z.enum(['pass', 'partial', 'fail']);

// --- Line-anchor discipline (D-04) --- //
// lineId format is ${fileId}:h${hunkIdx}:l${lineIdx} per Phase 1 D-17.
// A regex at the schema boundary kills freeform (path, line) hallucination at ingress.
const LineId = z.string().regex(
  /^[A-Za-z0-9_-]+:h\d+:l\d+$/,
  {
    error:
      'Invalid lineId. Every finding must anchor via an opaque `lineId` returned by `get_hunk` or `list_files`. Format: `<fileId>:h<hunkIdx>:l<lineIdx>`. Never supply `path:line` strings.',
  }
);

// --- Finding schema --- //
const Finding = z.object({
  category: Category,
  checklistItemId: z.string().min(1),
  severity: Severity,
  lineId: LineId,
  title: z.string().min(1).max(200),
  rationale: z.string().min(1).max(2_000),
});

// --- Top-level tool input --- //
// .refine on the findings array is how D-03 (nit cap) lives in the schema.
// .default('request_changes') on verdict is how D-05 (default inversion) works.
const Input = z.object({
  findings: z.array(Finding).refine(
    (fs) => fs.filter((f) => f.severity === 'nit').length <= 3,
    {
      error:
        'Too many nits (>3). The nit cap is structural — promote the most important issue to `minor` or higher, and drop the rest. See tool description for nit discipline.',
    }
  ),
  coverage: z.object({
    correctness: CoverageStatus,
    security: CoverageStatus,
    tests: CoverageStatus,
    performance: CoverageStatus,
    style: CoverageStatus,
  }),
  verdict: Verdict.default('request_changes'),
  prKey: z.string().min(1),
});

// Build the adversarial-framing description at server-start time, pulling the
// TS-const checklist inline. The Claude Code session reads this verbatim via
// `tools/list` — it IS the prompt for this tool. No runtime templating.
const DESCRIPTION = [
  'Produce an adversarial self-review of the PR under review. YOUR JOB IS TO FIND REASONS TO REQUEST CHANGES, not to approve.',
  '',
  'Default verdict is `request_changes`. You must ARGUE THE VERDICT DOWN to `comment` or `approve` with evidence; never drift UP from `approve`. If you find no meaningful issues after a thorough pass, THEN downgrade to `comment` or `approve`.',
  '',
  'Checklist (criticality-ranked):',
  ...CHECKLIST.map(
    (item) => `  [${item.category}:${item.id}] ${item.text}${item.evaluationHint ? ` — ${item.evaluationHint}` : ''}`
  ),
  '',
  'Devil\'s advocate pass — before submitting, ask:',
  '  - What could break at null / empty / error / concurrent boundaries?',
  '  - What did the author forget (tests, rollback, telemetry, edge cases)?',
  '  - If I authored a summary via set_pr_summary, does the intent lens apply here? (refactor → behavior preservation; feature → correctness+tests; bug-fix → regression check)',
  '',
  'Nit discipline: MAX 3 findings of severity `nit`. More than 3 and the schema rejects the entire payload. Promote the most important, drop the rest.',
  '',
  'Anchors: every finding must reference a `lineId` returned by `list_files`/`get_hunk`. Never fabricate path:line strings.',
].join('\n');

export function registerRunSelfReview(mcp: McpServer, manager: SessionManager): void {
  mcp.registerTool(
    'run_self_review',
    { title: 'Run Self-Review', description: DESCRIPTION, inputSchema: Input.shape },
    async ({ findings, coverage, verdict, prKey }) => {
      try {
        // Server-side resolve lineId → (path, line, side). Unknown lineIds
        // throw a structured error the catch block turns into isError:true.
        const resolved: ResolvedFinding[] = resolveFindings(manager, prKey, findings);
        const selfReview: SelfReview = {
          findings: resolved,
          coverage,
          verdict,
          generatedAt: new Date().toISOString(),
        };
        // Atomic silent replace via the Phase-2 reducer pipeline. per-prKey
        // promise-chain queue serializes; SSE fanout pushes to the browser.
        await manager.applyEvent(prKey, { type: 'selfReview.set', selfReview });

        // Tool return is a brief ack — the *effect* is the SessionEvent above.
        // Stays well under the ~2k-token cap even for 25-finding payloads.
        return {
          content: [{ type: 'text' as const, text: renderAck(selfReview) }],
        };
      } catch (err) {
        logger.error('run_self_review failed', err);
        // isError:true makes this LLM-visible — the Claude Code session sees
        // the text and can self-correct. Thrown = protocol error = hidden.
        return {
          content: [{ type: 'text' as const, text: renderCorrective(err) }],
          isError: true,
        };
      }
    }
  );
}
```

**Tool Use:**

Phase 4's tool surface is exactly four tools — `list_files`, `get_hunk`, `set_pr_summary`, `run_self_review` (D-14). They split cleanly along read/write:

- **Read tools** (`list_files`, `get_hunk`) return diff data from `manager.getSession(prKey)`. No state mutation, no `applyEvent` call. Their response payload IS the value (unlike write tools). Both enforce the ~2k-token response cap via pagination — `list_files` cursor-paginates at 30 files/page (D-15); `get_hunk` slices within a hunk via cursor (D-16).
- **Write tools** (`set_pr_summary`, `run_self_review`) mutate `ReviewSession` via `manager.applyEvent(prKey, { type: 'summary.set' | 'selfReview.set', ... })`. Return a brief ack string only. The state delta propagates to the browser via the existing Phase-2 SSE `UpdateMessage` fanout — no new HTTP endpoint, no new WebSocket, no new client action beyond `onSummarySet` / `onSelfReviewSet` in `web/src/store.ts`.

Registration is a four-line addition to `server/src/mcp/server.ts`:

```ts
// server/src/mcp/server.ts (post-Phase-4)
import { registerStartReview } from './tools/start-review.js';
import { registerListFiles } from './tools/list-files.js';
import { registerGetHunk } from './tools/get-hunk.js';
import { registerSetPrSummary } from './tools/set-pr-summary.js';
import { registerRunSelfReview } from './tools/run-self-review.js';

// ...inside startMcp()...
registerStartReview(mcp, manager);
registerListFiles(mcp, manager);
registerGetHunk(mcp, manager);
registerSetPrSummary(mcp, manager);
registerRunSelfReview(mcp, manager);
```

No `outputSchema` is declared on any Phase-4 tool. The SDK supports it (it produces `structuredContent` in the result alongside `content`), but Phase 4's write tools return a human-readable ack and Phase 4's read tools return pre-formatted text; structuredContent would duplicate without value. Revisit if a Phase-5 client needs structured read-tool results.

**State Management:**

Phase 4 piggybacks entirely on the Phase-2 event-sourcing pipeline — no new infrastructure.

- **Events:** Two new `SessionEvent` union branches in `shared/types.ts`:
  - `{ type: 'summary.set', summary: PrSummary }`
  - `{ type: 'selfReview.set', selfReview: SelfReview }`
- **Reducer:** Two new case branches in `server/src/session/reducer.ts`, both one-liners:
  - `case 'summary.set': return { ...s, summary: e.summary };`
  - `case 'selfReview.set': return { ...s, selfReview: e.selfReview };`
- **Manager:** No changes to `applyEvent` itself — it owns `lastEventId` increment (Phase 2 Plan 02-03 grep-enforced invariant) and the per-`prKey` Promise-chain queue. Phase 4 mutations inherit serialization + SSE fanout for free.
- **Session fields:** `ReviewSession` gains `summary?: PrSummary | null` and `selfReview?: SelfReview | null`. Both optional so pre-Phase-4 snapshots on disk load cleanly (zero migration).
- **Resolution step (findings):** `run_self_review`'s handler resolves each `Finding.lineId` to `(path, line, side)` against `session.diff` BEFORE emitting the `selfReview.set` event. The resolved triplet is stored alongside the `lineId` on each `ResolvedFinding` so the UI renders `file:line` without re-resolving and so downstream phases (Phase 5 `draft_comment`) can key off the server-resolved position.
- **Silent replace discipline:** Both events are unconditional overwrites. No history array, no version flip-back, no `stale: true` flag (D-07, D-08). If the diff changes via `session.adoptNewDiff`, the summary/self-review stays and the user regenerates manually.

**Context Window Strategy:**

The Claude Code session's context budget is the constrained resource — not the plugin's. The plugin's obligation is **keep tool responses lean and tool-call count low**.

- **Per-tool response cap: ~2k tokens (Pitfall 5).** `list_files` pages at 30 files; `get_hunk` paginates within-hunk via cursor; `set_pr_summary` / `run_self_review` acks are one-paragraph render strings.
- **Tool count cap: ~10 tools total (Pitfall 15).** Phase 4 ends at 5/10 (D-14). Planners re-verify at each phase boundary.
- **Description budget: effectively uncapped but proportional.** The adversarial-framing description for `run_self_review` is the biggest legitimate consumer (~400 words). That cost is paid once per session (at `tools/list` time) — it does NOT recur per tool call.
- **No server-side summarization, no truncation, no chunking** inside the plugin. The Claude Code session manages its own compaction via Claude Code's built-in context-management; the plugin stays out of that loop. If a repeated pattern emerges (e.g., the session re-fetches the same hunk unnecessarily), add a client-side cache hint — not server-side eviction.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice. Template examples are Python/Pydantic — this project is TypeScript, so examples below are TypeScript/zod equivalents throughout.

### Structured Outputs with Zod (the TypeScript equivalent of Pydantic)

In the usual "framework calls an LLM" topology, `zod` (or Pydantic in Python) wraps the LLM output and validates on the way back. In Phase 4's *inverted* topology, `zod` sits on the LLM *input* — the Claude Code session is the output producer, and the plugin's tool `inputSchema` is the contract the session must satisfy. This flips the retry mechanism: it's the **MCP protocol's `isError: true` surface**, not a framework retry loop.

**Phase-4 output schema (what the Claude Code session must produce for `run_self_review`):**

```ts
// shared/types.ts — exported for both server and web consumers.
import { z } from 'zod';

export const Severity = z.enum(['blocker', 'major', 'minor', 'nit']);
export const Category = z.enum(['correctness', 'security', 'tests', 'performance', 'style']);
export const Verdict = z.enum(['request_changes', 'comment', 'approve']);

export const Finding = z.object({
  category: Category,
  checklistItemId: z.string().min(1),
  severity: Severity,
  lineId: z.string().regex(/^[A-Za-z0-9_-]+:h\d+:l\d+$/),
  title: z.string().min(1).max(200),
  rationale: z.string().min(1).max(2_000),
});

export const SelfReviewInput = z.object({
  findings: z.array(Finding).refine(
    (fs) => fs.filter((f) => f.severity === 'nit').length <= 3,
    { error: 'Too many nits (>3). Promote or drop.' }
  ),
  coverage: z.object({
    correctness: z.enum(['pass', 'partial', 'fail']),
    security:    z.enum(['pass', 'partial', 'fail']),
    tests:       z.enum(['pass', 'partial', 'fail']),
    performance: z.enum(['pass', 'partial', 'fail']),
    style:       z.enum(['pass', 'partial', 'fail']),
  }),
  verdict: Verdict.default('request_changes'),
});

export type SelfReviewInput = z.infer<typeof SelfReviewInput>;
```

**How the framework integrates** (MCP SDK specific): `McpServer.registerTool` accepts the raw zod shape (`SelfReviewInput.shape`). When the Claude Code session calls the tool, the SDK parses the arguments against the shape internally and:
- on success → invokes the handler with the typed, validated input;
- on failure → returns a JSON-RPC error to the client *and* surfaces the validation issues in a structure the Claude Code session reads.

The LangChain equivalent would be `.with_structured_output(SelfReviewInput)`; the Pydantic-Python equivalent would be `instructor`-wrapped OpenAI/Anthropic clients. **None of these apply here**, because this plugin does not call any LLM. The zod schema + MCP tool surface IS the structured-output boundary.

**Retry logic:**

- **How many retries:** not the plugin's call. The Claude Code session handles retry internally when it sees `isError: true` (or a JSON-RPC validation error). Empirically it retries once or twice, then surfaces the failure to the human user.
- **What the plugin must do:** write schema-rejection error messages that are **corrective, not terminal**. "Invalid input" is terminal; "Too many nits (>3). Promote the most important to `minor` or drop the rest." is corrective — the session can act on it. Every `.refine()` call and every validation-path `isError: true` response must include both (a) what failed and (b) the shape of the fix.
- **What to log:** `logger.error('run_self_review failed', err)` on the stderr channel. The MCP JSON-RPC channel is stdout-only; never `console.log` (pitfall #3 in Section 3).
- **When to surface to the human:** never from the plugin directly. The plugin's UI surface for failures is the absence of a summary/self-review blob in session state — the empty state (D-07: "Summary not generated yet — ask Claude to summarize this PR"). The Claude Code session decides when to escalate to the user.

### Async-First Design

Every MCP tool handler is `async` by signature and `await`ed by the SDK. For Phase 4 this is mostly trivial — the handlers are in-memory operations on `SessionManager`.

- **How async works here:** Handlers return `Promise<CallToolResult>`. The SDK serializes JSON-RPC requests per connection, so handlers don't race each other at the transport layer. State mutations go through `manager.applyEvent` which is itself `async` and runs inside the per-`prKey` Promise-chain queue (Phase 2) — that queue IS the concurrency-control primitive; don't add new locks in tool handlers.
- **The one common mistake** (TypeScript / Node / MCP-specific): **not `await`ing `manager.applyEvent`**. The call returns a promise that resolves only after the reducer has run, the snapshot has been persisted, and SSE has fanned out to browser clients. Forgetting the `await` makes the tool handler return its ack *before* the state actually lands — the browser gets the update milliseconds after the Claude Code session sees the ack, and any subsequent tool call that reads state races the write. Always `await manager.applyEvent(...)` before `return { content: ... }`.
- **Streaming vs. await:** Phase 4 is all `await`. No tool streams — `run_self_review` is atomic (D-01, rejected lifecycle variant); `set_pr_summary` is atomic; `list_files` / `get_hunk` return a single paged response. Reactivity in the browser comes from the Phase-2 SSE `UpdateMessage` fanout *after* the tool returns — the streaming surface is the existing SSE channel, not the MCP protocol.
- **Node 22 ESM gotcha:** all handler files end `.ts` but import siblings as `.js` (`import './foo.js'`) per `"module": "Node16"` ESM rules. TypeScript compiles `.ts` → `.js` and the runtime import resolves at `.js`. This is already the convention in `start-review.ts`; deviating breaks the MCP subpath imports.

### Prompt Engineering Discipline

Phase 4's "prompt" lives in the tool `description` strings — D-20 is the project-specific statement of "prompt engineering discipline." There is no system-prompt vs. user-prompt split because there is no LLM API call; the Claude Code session reads the description via `tools/list` at connection time.

- **"System vs. user prompt separation" maps to `description` vs. `inputSchema`.** The description is the *persistent* framing the session sees for this tool — role ("adversarial reviewer"), criticality checklist, nit cap, default-verdict inversion. The inputSchema is the *per-call* shape the session must produce. Keep role/persistent framing in the description; keep per-call data in the schema. Never try to inject per-call prompting through the inputSchema (e.g., a `.default('...instructions...')` on a freeform string field) — that confuses the schema/prompt boundary.
- **Few-shot guidance: inline, not dynamic.** D-20 is explicit: no runtime prompt templating, no separate prompt file, no server-owned system prompt. If a few-shot example is needed to pin nit discipline or verdict-inversion intuition, it goes **inline in the tool description string** at server-start time, built from the TS-const checklist (D-02). That way the session sees the same framing every call and there's no prompt-drift between invocations.
- **Intent → review-lens coupling is soft, not schema-level** (D-21). `run_self_review` does not take `intent` as an input parameter; the description instructs "If you authored a summary via `set_pr_summary`, apply the intent-appropriate review lens (refactor → behavior preservation; feature → correctness+tests; bug-fix → regression check)." Deliberately avoids forcing a call order.
- **No `max_tokens` surface exists** because the plugin makes no LLM call. The analogous discipline is: **the tool description must stay concise per tool** (proportional — a 400-word `run_self_review` description is fine because adversarial framing is its job; a 400-word `get_hunk` description would be wasteful). And **every response text field must have a zod `.max()` length** (`title: z.string().max(200)`, `rationale: z.string().max(2_000)`) so the session cannot dump unbounded prose into a finding and bust the ~2k-token response cap on the UI-mutation path. Bounded strings are Phase 4's equivalent of "set max_tokens explicitly in production."
- **Paraphrase-fidelity enforcement is social, not mechanical** (D-09). The `set_pr_summary` description asks the session to paraphrase the PR description faithfully; the UI renders `paraphrase` adjacent to a collapsed "Author's description" pane so mismatches are visible. No Jaccard-overlap check, no length floor — those are fragile. The deterministic `paraphrase()` helper in `start-review.ts` exists as a belt-and-suspenders sanity baseline, available for the `set_pr_summary` handler to optionally cross-check against if the planner decides it's worth it (not required by D-09).

### Context Window Management

Phase 4 pays Pitfall 5's BLOCKER cost explicitly. The "context window" under management here is the **Claude Code session's own context budget**, not an LLM API request context — but the mechanics are the same: big responses starve subsequent tool calls.

- **`list_files` is the pagination entry point.** D-15 caps at 30 files/page default; the response carries `{ files, nextCursor, totalFiles, excludedCount }` so the session knows when to stop. Generated files (Phase 3 D-14's `generated: true` flag) are filtered by default; `includeExcluded: true` opts in. Opaque cursor (planner picks base64 offset or similar).
- **`get_hunk` slices within-hunk.** D-16 paginates inside a single hunk via cursor, carrying `{ hunkId, fileId, path, header, lines, nextCursor, totalLines }`. Normal hunks fit in one call (`nextCursor: null`); a 400-line snapshot-test diff pages through. Each `DiffLine` carries its opaque `lineId` so the session can cite findings without re-fetching.
- **No server-side compaction or summarization.** The plugin does not own Claude Code's context-management loop. Phase-4's contribution is keeping every response under the ~2k-token ceiling so the session has headroom to keep the rest of its context.
- **Findings responses are lean.** `run_self_review` returns a one-paragraph ack; the full resolved findings blob lives in session state (and on the browser via SSE), not in the tool response. That's the inversion-of-topology insight paying off: the session doesn't need to re-read what it just wrote; the UI does.
- **`run_self_review` does not read its own prior state.** Regeneration is atomic silent replace (D-01 / D-08) — the session doesn't fetch the previous `selfReview` as a baseline. If that pattern emerges in daily use as a UX gap, add a `get_self_review` tool in Phase 7; out of Phase-4 scope.

### Cost and Latency Budget

No model costs. No API quota. No tokens billed. The plugin makes zero LLM API calls (PROJECT.md constraint). The costs Phase 4 governs are **latency** and **context-window consumption on the Claude Code session side**.

- **Per-call latency estimate:** every Phase-4 tool is in-memory work on `SessionManager` state (reads) or a single serialized `applyEvent` (writes). No disk I/O beyond the Phase-2 atomic snapshot write (~sub-5ms on SSD), no network round-trip (no GitHub calls, no Anthropic calls). P99 latency per tool call is well below any Claude-Code-side MCP timeout (Pitfall 7 mitigation — nothing in Phase 4 can block past the default 60s).
- **Per-call "cost" in context budget:** `list_files` ≈ 30 files × ~30 tokens = ~900 tokens; `get_hunk` ≈ one hunk ≈ 500-1500 tokens; `set_pr_summary` ack ≈ 50 tokens; `run_self_review` ack ≈ 50 tokens. A full review session (enumerate + ~10 hunk fetches + summary + self-review) lands around 10-20k tokens on the Claude Code side — well inside the session's budget.
- **Caching:** the Phase-2 snapshot IS the cache. Reading from `manager.getSession(prKey)` is O(1); both read tools go through that path with no additional memoization. No Redis, no in-process TTL cache. If profiling reveals a hot-path cost (unlikely at single-user scale), add a `WeakMap`-keyed cache inside `get-hunk.ts` — not a new infrastructure layer.
- **Cheaper-models-for-sub-tasks** is not applicable — there's only one "model" in the loop and it's whatever the user configured in Claude Code. Phase 4's analogous optimization is **static data where possible**: the `CHECKLIST` const is a TS value, not a file read on every server start; the `DESCRIPTION` strings are built once at `registerTool` time; the `paraphrase()` helper in `start-review.ts` is a deterministic textual transform (no LLM), available as a cheap cross-check for `set_pr_summary`'s paraphrase field if the planner wants the sanity rail (optional per D-09).

## 5. Evaluation Strategy

> Framed around the five rubric ingredients from Section 1b, each of which maps 1-to-1 to a ROADMAP success criterion. Section 1b explicitly flagged three constraints this strategy has to honor: (a) sample-size-of-one with strong confirmation-bias risk, (b) blind-labeled fixtures to offset user's in-the-moment priors, (c) LLM-as-judge for mechanical dimensions plus regression anchoring for the rest. This section operationalizes all three.
>
> **Topology reminder (load-bearing for what follows):** the plugin makes **zero LLM API calls** at runtime. The subject-under-test in every eval dimension below is the Claude Code session's output shape when it calls `set_pr_summary` / `run_self_review` — not a model the plugin is invoking. That means:
> - There is no online LLM-as-judge. The plugin has no API key and no model provider; it literally cannot score its own traffic in production.
> - Every dimension that requires semantic judgment (signal ratio, paraphrase fidelity) is evaluated **offline** in a dev-time harness that drives fixture PRs through the plugin and captures the real Claude Code session's responses. The harness is run manually by the user before a release tag, not on every commit.
> - Every dimension that is mechanically checkable (anchor resolution, nit cap, verdict enum, coverage breadth, schema conformance) is checked **in-process inside the tool handler** at runtime AND asserted by vitest against the fixture corpus — no LLM judge needed.

### Dimensions

Five dimensions, one per rubric ingredient, one per ROADMAP success criterion. Each anchored to a fixture-based pass/fail test so regressions are caught mechanically.

| # | Dimension | Rubric (PASS / FAIL / Measurement) | Priority |
|---|-----------|------------------------------------|----------|
| 1 | **Signal ratio** — useful findings over total findings (Section 1b ingredient 1; ROADMAP criterion 3) | **PASS:** On a fixture PR with N seeded defects, ≥60% of emitted findings are real-defect mappings (each finding's `rationale` names a seeded defect OR describes a concrete observable failure in the diff). Nits are ≤3 (schema-enforced by D-03). <br> **FAIL:** <60% of findings map to seeded defects; OR more than 3 nits land (impossible — zod rejects); OR the majority of findings are Tier-3 noise ("consider extracting", "naming could be clearer") with no evidence per-finding. <br> **Measurement:** LLM judge (dev-time only). The offline harness feeds each `(finding.title, finding.rationale, finding.lineId → diff_excerpt)` triple to an Anthropic-API-invoking judge script and classifies each finding as Tier-1/2 (signal) or Tier-3 (noise). Judge prompt is versioned in-repo; calibrated against a human-labeled subset of the fixture corpus before thresholds are trusted. Pass threshold: ≥60% signal per the Jet Xu 2025 baseline. | **Critical** |
| 2 | **Anchor correctness** — file:line refs point at real diff lines (Section 1b ingredient 2; ROADMAP criterion 2) | **PASS:** 100% of `finding.lineId` values in every fixture-run payload resolve to a real `(path, line, side)` triplet against `session.diff` via the handler's `resolveFindings()` path. Every finding that claims "this line does X" has a `lineId` pointing to a diff line whose content matches the rationale's claim (spot-checked by LLM judge on the fixture corpus). <br> **FAIL:** Any `lineId` fails the D-04 regex at the schema boundary; OR any `lineId` passes the regex but cannot be resolved (no matching `fileId`/`hunkIdx`/`lineIdx`); OR a resolved triplet points at a line whose content contradicts the rationale (e.g., rationale says "this null check" but the line is an import). <br> **Measurement:** Code-based for the first two (regex + resolver are both already on the handler path — `describe('lineId resolution')` vitest block asserts every fixture emits only resolvable IDs); LLM judge for the third (content-vs-rationale alignment), same harness as Dimension 1. | **Critical** |
| 3 | **Verdict calibration** — `request_changes` on genuine-bug fixtures, no inflation (Section 1b ingredient 3; ROADMAP criterion 5, verbatim) | **PASS:** On each of the three genuine-bug fixtures, `selfReview.verdict === 'request_changes'` AND at least one finding of severity `blocker` or `major` references the seeded bug (anchored to the `lineId` covering the bugged line). On each of the two benign-refactor fixtures, verdict is `approve` or `comment` with zero `blocker`s and zero findings falsely alleging a defect. <br> **FAIL:** Any genuine-bug fixture produces `verdict: 'approve'` or `verdict: 'comment'` (even with blocker findings present — that's verdict inflation). Any genuine-bug fixture produces `request_changes` but with no blocker/major finding in the bug's category (the bug is missed). Any benign-refactor fixture produces `request_changes` (false negative inverted — the reviewer invented a defect). <br> **Measurement:** Code-based. Each fixture carries a labeled `expected = { verdict, bugLineId?, bugCategory? }` alongside its PR body + diff. Vitest asserts the emitted `selfReview` satisfies the expected shape. No judge needed — the fixture IS the ground truth. | **Critical** |
| 4 | **Coverage breadth** — all 5 checklist categories touched (Section 1b ingredient 4; ROADMAP criterion 4) | **PASS:** On every fixture, `selfReview.coverage` carries a non-null status (`pass`/`partial`/`fail`) for every one of the five categories (correctness, security, tests, performance, style). At least one finding OR one explicit `pass` claim justifies each category status. <br> **FAIL:** Any of the five coverage keys is missing (impossible — zod rejects; the schema is exhaustive by enum), OR a category is marked `pass` with no findings AND no rationale-supported claim (the coverage chip is green but the reviewer never looked). <br> **Measurement:** Code-based for the exhaustive-key check (zod's `z.object({correctness, security, tests, performance, style})` already enforces it). Code-based heuristic for the justification check: for every category where `coverage[cat] === 'pass'` AND no finding has `category === cat`, the eval flags it as "unjustified pass" and asserts that ≤ 1 of 5 categories is in that state per fixture. A fixture where 3 categories are unjustified-pass fails the eval. | **High** |
| 5 | **PR-description fidelity / intent classification** — paraphrase preserves author framing; intent matches fixture label (Section 1b ingredient 5; ROADMAP criterion 1) | **PASS:** `summary.intent` equals the fixture's labeled intent enum value (`bug-fix` / `refactor` / `feature` / `chore` / `other`). Paraphrase is judged (LLM-as-judge) to preserve the author's framing (keeps the author's stated goal intact; does not re-summarize the diff). On the intent-mismatch-trap fixture, `intent` matches the **code's actual** intent, not the PR description's stated intent, AND `summary.intentConfidence` is below 0.8 (reflecting the authorial ambiguity). <br> **FAIL:** Intent wrong on any non-trap fixture. Paraphrase abstracts the patch instead of restating the author. On the trap fixture, intent is copied verbatim from PR description with high confidence (the session didn't notice the mismatch). <br> **Measurement:** Code-based for the intent enum match (strict equality). LLM judge for paraphrase fidelity (calibrated against a human-labeled subset of the paraphrases before trusting). Confidence threshold on the trap case is a code-based numeric assertion. | **High** |

**Calibration & confirmation-bias mitigations (Section 1b requirement):**

1. **Blind labeling:** Four of the ten fixtures are "blind" — the user authors the diff + seed defect against only the published rubric sources (Jet Xu signal tiers, Google eng-practices must-fix vs. optional, CodeRabbit severity taxonomy), then does not look at the diff again until eval time. The labeled `expected` shape for these four fixtures MUST be approved by walking through the published-rubric reasoning in a comment block in the fixture file. Personal-taste overrides are not allowed on blind fixtures. This is how the sample-size-of-one problem gets a second opinion without a second person.
2. **LLM judge calibration:** Before the first production release, the user hand-labels every finding emitted across the full fixture corpus (expected maybe 60-120 findings across 10 fixtures) as Tier-1/Tier-2/Tier-3. The LLM judge prompt is then tuned against that hand-labeled set until judge-vs-human agreement reaches ≥0.7 correlation (per the `ai-evals.md` guidance). After calibration the judge is trusted for repeat runs; re-calibration required if the judge prompt or the dimension rubric changes.
3. **Regression anchoring:** Once a fixture passes an eval run, its `expected` shape is frozen in-repo. Any later change that breaks an already-passing fixture is a hard failure regardless of in-the-moment taste. The user cannot "update the rubric" to paper over a regression — they can either fix the underlying behavior or explicitly mark the fixture deprecated with a comment block explaining why the frozen expectation no longer applies.

## 6. Guardrails

> Most online guardrails for Phase 4 are **structural and already in place** via zod schemas — D-03 nit cap, D-04 lineId regex, D-05 verdict default, the severity/category/coverage-status enums. This section covers what runtime enforcement **beyond zod** is needed, and what goes on the offline-flywheel side for continuous improvement.

### Online (Real-Time) Guardrails

Runs inside the MCP tool handler on every `set_pr_summary` / `run_self_review` invocation. Each adds latency, so the list is minimal — every entry has to earn its place.

| # | Guardrail | Trigger | Intervention |
|---|-----------|---------|--------------|
| G-1 | **Zod schema rejection with corrective message** (already structural; listed here for completeness — this is the workhorse) | Any zod validation failure: severity enum miss, verdict enum miss, nit cap breach, malformed lineId, missing coverage key, title/rationale length overflow. | **Block** — `isError: true` with a corrective-shape message per Pitfall #2 in Section 3. The Claude Code session reads the text and self-corrects within its own retry loop. Never `throw` — thrown exceptions become protocol-level errors the session can't recover from. |
| G-2 | **lineId resolution gate** (post-zod, pre-reducer) | A lineId passes the D-04 regex but fails to resolve against `session.diff` (no matching `fileId`, out-of-range `hunkIdx`, out-of-range `lineIdx`). | **Block** — handler returns `isError: true` with the corrective text: *"`lineId` X does not resolve to a diff line in this review session. Re-fetch the current hunk list via `list_files`/`get_hunk` and anchor findings against those IDs."* Prevents the "finding references a file the session hallucinated" failure mode from landing in `session.selfReview`. |
| G-3 | **Unjustified-pass coverage sanity check** (post-resolution, pre-reducer) | `run_self_review` payload has 3+ categories set to `pass` in `coverage` but zero findings in any of those categories AND zero explicit-pass rationale text. Threshold: >2 unjustified-pass categories flagged per payload. | **Flag (not block)** — handler emits a warning `SessionEvent` (`{ type: 'selfReview.qualityWarning', code: 'unjustified_pass', categories: [...] }`) that the UI renders as a muted banner in the StageStepper band. Does NOT reject the payload (would be too aggressive — some refactors legitimately have 3 pass categories). Downstream: the flywheel metric (F-1 below) counts these warnings over time. |
| G-4 | **Rationale-vs-lineId content-cohesion drift detector** | (Optional, planner's call on whether to ship in Phase 4 vs. defer to Phase 7) A finding's rationale mentions a token (`"null check"`, `"auth"`, `"missing test"`) that does not appear in any of the ±3 diff lines around the resolved `(path, line)` of its `lineId`. Purely heuristic; high false-positive rate. | **Flag (not block)** — same `selfReview.qualityWarning` event with `code: 'rationale_lineid_mismatch'`. Flagged findings render with a muted caution icon in the sidebar. The user's own eyeball is the last-line check; the plugin is not in the business of silently dropping findings. **Default for Phase 4: NOT implemented.** Ship only if the first week of daily use shows the content-mismatch failure mode actually biting. |

Guardrails G-1 and G-2 are **mandatory** — they are the non-zod-level structural enforcement the plugin must have to be trustworthy. G-3 is **recommended** and low-cost. G-4 is **explicitly deferred** unless evidence accrues.

### Offline (Flywheel) Metrics

Runs as part of the `test:eval` harness (manual pre-release) AND as a background sampling pass on real session state (see Section 7 sampling). Feeds continuous-improvement signal back into the tool descriptions, fixture corpus, and/or checklist wording.

| # | Metric | Sampling Strategy | Action on Degradation |
|---|--------|-------------------|----------------------|
| F-1 | **Unjustified-pass rate** — fraction of real self-reviews emitting a `selfReview.qualityWarning` with `code: 'unjustified_pass'` | Every real session's self-review is logged to the Section 7 event log; the flywheel pass counts warnings over a rolling 20-review window. | If >25% of recent real reviews trip the warning, the user updates the `run_self_review` description to re-emphasize "every `pass` chip must be backed by a finding or rationale claim." |
| F-2 | **Sycophancy-drift over calendar time** — re-run fixture 01 (null-pointer bug) periodically; track whether verdict, signal ratio, and coverage breadth scores drift month-over-month | Monthly (or when the Claude Code model changes under the hood — Anthropic ships a new model, the user notices a behavior shift). The user re-runs `pnpm test:eval` and diffs the output against the last committed baseline (checked into `server/src/__tests__/evals/baselines/` at each release tag). | If fixture 01 starts passing with a `comment` verdict (regression from `request_changes`), or signal ratio drops >10 points, the issue is captured as a Phase-7 polish item (description re-tightening) or a STATE.md note for the next phase-kickoff retrospective. |
| F-3 | **Nit-cap boundary pressure** — distribution of `nit` count across real self-reviews (do sessions consistently pack exactly 3 nits? exactly 0?) | Every real self-review payload's `nit` count is logged. Flywheel pass computes the histogram over a rolling 50-review window. | If >40% of payloads consistently hit exactly 3 nits, that's evidence the session is treating the cap as a quota and padding. Update the description to frame nits as "omit unless load-bearing" rather than "up to 3 permitted." |
| F-4 | **Intent classification distribution** — which intent the session assigns across all real sessions | Every `summary.intent` emission is logged. Flywheel pass over 20-review window checks whether any intent is suspiciously over-represented. | If >60% of sessions classify as `other`, that's a sign the fixed enum doesn't fit the user's real PR shapes — candidate for enum expansion in Phase 7. If `feature` is never assigned, that's the opposite signal (intent is being collapsed away). |
| F-5 | **Schema-rejection corrective-retry rate** — fraction of `run_self_review` attempts that zod-rejected on first call and succeeded on retry | Server-side logger: count `isError: true` returns keyed by `(prKey, tool, attemptNumber)` in the event log. Flywheel over rolling 50-attempt window. | If >20% of real sessions require a retry, the corrective-message text isn't guiding the session well enough. Tighten the `.refine(...)` error message or the description's nit-cap copy. If >5% require a second retry (two failures in a row), that's a blocker — the session is stuck in a loop and the cap/enum wording needs rethinking. |

## 7. Production Monitoring

> "Production" here is the solo-dev user running the plugin in Claude Code against their own PRs. There is no multi-tenant surface, no on-call rotation, no user pager. "Monitoring" means: what file-based artifact lets the user (or a future audit pass) retrospectively answer *"did self-review quality drift this month?"* without reading every self-review manually. The emphasis is on **post-hoc analysis**, not real-time alerting.

**Tracing tool: structured JSONL event log at `${CLAUDE_PLUGIN_DATA}/event-log.jsonl` + existing `logger.error` stderr stream.**

Per the required-reading detection scan (`grep -r "langfuse\|langsmith\|arize\|phoenix\|..."` returned only planning-doc references, nothing installed), no tracing infrastructure exists. Arize Phoenix is the default in `ai-evals.md`, but its value proposition — capturing LLM spans with prompts/completions/token counts via OpenTelemetry — is zero for a plugin that never emits an LLM span. The span surface available is MCP-tool-invocation events; Phoenix over that is ceremony without signal (see Section 5 for the longer rationale). A structured JSONL log gives the same post-hoc analysis surface without the operational overhead.

**Event log schema** — one JSON object per line, append-only, atomic writes via `write-file-atomic` (already a dep). Events written from the MCP tool handlers via a tiny `eventLog.record(...)` helper mounted on the `SessionManager`.

```ts
// shared/types.ts — exported from the events module.
type EventLogRecord = {
  ts: string;                          // ISO timestamp
  prKey: string;                       // owner/repo/PR#  or branch-diff key
  tool: 'set_pr_summary' | 'run_self_review' | 'list_files' | 'get_hunk';
  outcome: 'ok' | 'zod_reject' | 'resolver_reject' | 'quality_warning';
  attemptNumber: number;               // 1 = first call; 2+ = after corrective retry (F-5)
  // Tool-specific envelope — bounded in size so the log never explodes.
  payloadFingerprint: {
    // For set_pr_summary:
    intent?: 'bug-fix' | 'refactor' | 'feature' | 'chore' | 'other';
    intentConfidence?: number;
    paraphraseLen?: number;
    // For run_self_review:
    findingCount?: number;
    nitCount?: number;
    severityHistogram?: Partial<Record<'blocker'|'major'|'minor'|'nit', number>>;
    coverageHistogram?: Partial<Record<'pass'|'partial'|'fail', number>>;
    verdict?: 'request_changes' | 'comment' | 'approve';
    qualityWarningCodes?: ('unjustified_pass' | 'rationale_lineid_mismatch')[];
  };
};
```

Deliberately does **not** log full finding text, paraphrase bodies, or diff line contents. Reasons: (a) those are already on disk in `state.db` per session — the event log is a *derived* time-series, not a second copy of the authoritative state; (b) keeps the log file small enough that the flywheel pass can load it into memory at once; (c) every field is a fingerprint aggregate, which is exactly what the flywheel metrics in Section 6 consume.

**Key metrics to track:**

1. **Verdict distribution per fixture+calendar-month** (Dimension 3 regression signal) — percentage of self-reviews landing at each verdict, bucketed. A silent shift from `request_changes`-majority to `approve`-majority across real sessions over a month is the early sign of sycophancy drift.
2. **Nit-count histogram** (F-3 flywheel) — distribution of the `nitCount` field across the rolling 50-review window.
3. **Schema-rejection corrective-retry rate** (F-5 flywheel) — fraction of `outcome: 'zod_reject'` entries followed by `outcome: 'ok'` on the same `prKey`+`tool` within 60s.
4. **Resolver-rejection rate** (G-2 frequency) — fraction of `outcome: 'resolver_reject'`. Should trend toward zero; a non-zero trend is a sign the session is not using `get_hunk` output as the ID source-of-truth.
5. **Quality-warning rate** (G-3/G-4 frequency) — fraction of `run_self_review` records carrying non-empty `qualityWarningCodes`. Feeds F-1 in particular.

**Alert thresholds (what "drift" looks like — no pager, just retrospective thresholds the user watches):**

| Signal | Threshold | Response |
|--------|-----------|----------|
| Schema-rejection rate (F-5) | >20% over rolling 50-attempt window | Tighten the corrective error message or the description's cap/enum copy. Captured as a Phase-7 polish item; not a hotfix. |
| Double-retry rate (two rejections in a row on same `prKey`+`tool`) | >5% over rolling 50-attempt window | **Treat as a Phase-4 regression.** The corrective message isn't guiding the session out of the loop. Stop and redesign the message before next release. |
| Verdict-drift (rolling 20-review window) | `request_changes` share drops >15 percentage points vs. the Phase-4-release baseline (captured at fixture-01 pass time) | Re-run full eval harness. If fixture 01 now passes with `comment`, treat as sycophancy regression. |
| Resolver-rejection rate (G-2) | >5% of `run_self_review` attempts | Session is fabricating `lineId`s — means the description's "anchor via `get_hunk` output" guidance isn't biting. Re-run fixture 08 (anchor-trap) as regression check. |
| Quality-warning rate (G-3) | >25% of real reviews over rolling 20-window | Tighten coverage justification in description (F-1 action). |

No auto-page, no auto-rollback, no auto-disable. The response in every case is "next time the user runs `pnpm test:eval` or sits down to a Phase-7 polish block, they read these thresholds and respond." This is appropriate for a solo-dev tool where the user is also the operator.

**Smart sampling strategy:**

Every real MCP tool call appends an `EventLogRecord` — that's the **100% sampling at the fingerprint level**. Full payloads (finding text, paraphrase bodies) are **not** sampled into a separate store because they already exist in `state.db` for that session, indexed by `prKey`.

The flywheel sampling question — *which sessions deserve human re-read beyond the fingerprint?* — is answered with signal-based filters, not random sampling:

- **Always re-read (high-signal):** any session where `outcome === 'resolver_reject'`, any session where the same `prKey` shows two rejections in a row, any session where the user manually regenerated the self-review more than twice (regeneration count is trackable via `generatedAt` on `session.selfReview` across snapshots — optional; planner's call on exposing it via the event log).
- **Monthly sampling (calibration):** at the beginning of each month, the flywheel script samples 5 `outcome: 'ok'` sessions uniformly at random from the past 30 days and the user re-reads them against the Section 5 dimensions. Catches gradual drift the fingerprints miss (finding quality, paraphrase fidelity).
- **Baseline anchor:** at every release tag, `pnpm test:eval --write-baseline` overwrites `server/src/__tests__/evals/baselines/<version>.json` with the current fixture scores. Flywheel F-2 diffs against the most recent baseline.

Combined, this gives the user a **post-hoc analysis surface** (the JSONL log + baselines) rather than a **real-time monitoring surface** (which would be overkill for a solo tool). The event log is small — a full year of daily use at ~20 tool calls/day is ~7300 lines, well under 10 MB. No rotation, no retention policy, no external ingest needed.
