<decisions>
## Implementation Decisions

### Step-by-step content layout
- **D-01:** Summary step becomes its own full-page view — the summary card replaces the diff as the main content area, with the chat panel on the right. No diff is shown on the Summary step.
- **D-02:** Walkthrough, Review, and Submission steps keep the diff viewer as the main content area on the left, with step-specific content + chat on the right.
- **D-03:** The right panel in every step has step-specific content above and the chat input with suggested prompts pinned at the bottom.
- **D-04:** Chat context badge changes per step (SUMMARY, WALKTHROUGH + filename, REVIEW + filename + line, SUBMISSION + filename).
- **D-05:** WalkthroughStepList moves from a banner overlay to cards in the right panel above the chat.
- **D-06:** FindingsSidebar moves from a sidebar overlay to inline content in the Review step's right panel, ranked by severity.
- **D-07:** SubmitModal transforms from a modal into its own Submission step view — pre-submit checklist card + Claude's verdict suggestion in the right panel.

### Navigation + footer redesign
- **D-08:** Top bar splits into two rows: Row 1 = PR info (repo/number, title, branch arrows, CI badge, Settings, verdict button, Submit review). Row 2 = step navigation with numbered circles (1-4), labels, sub-labels (e.g., "6 steps · 4 reviewed"), connected by chevron arrows.
- **D-09:** Finding category badges (Correctness, Security, Tests, Performance, Style) move out of the navigation bar entirely — they appear in the Review step's right panel instead.
- **D-10:** Footer becomes contextual per step: Summary = "Regenerate summary" + "Last generated Xm ago" + "Continue to walkthrough →". Walkthrough = "Mark all reviewed" + "Continue to review →". Review = "Export as comments" + "Continue to submission →". Submission = no separate footer (CTA is in right panel).
- **D-11:** Keyboard hint bar is dropped entirely — no replacement needed.

### Claude's Discretion
- Exact spacing, border-radius, shadows, and micro-styling details to match design.html as closely as possible — use the design as the pixel reference.
- How to handle the "Ask" buttons shown per summary section (Why, What Changed, Risk) — these can be wired to send chat messages in the existing chat system.
- Suggested prompts content — use placeholder text matching the design's examples, contextual per step.

</decisions>

<specifics>
## Specific Ideas

- The design.html file at project root is the single source of truth for all visual decisions. Serve it locally and compare pixel-by-pixel.
- Each step in the design has a unique layout — do not use a generic template. Summary is fundamentally different (no diff) from the other three steps.
- The design shows "Claude's summary AUTO" vs "Author's description ORIGINAL" tabs on the Summary step — implement as tabs toggling between the two views.
- Stats grid on Summary step (files, added, removed, commits) should pull from existing PR metadata.
- "Regenerate summary" and "Last generated Xm ago" in footer use existing functionality from set_pr_summary / set_walkthrough MCP tools.

</specifics>
