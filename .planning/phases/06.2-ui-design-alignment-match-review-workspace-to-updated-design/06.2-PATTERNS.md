## Pattern Assignments

### `web/src/App.tsx` (component shell, event-driven)

**Analog:** `web/src/App.tsx` (current, lines 1-547)

The current App has a flat layout: TopBar + StageStepper + main (FileExplorer | DiffViewer | optional overlays). The redesign makes it step-routed. The key structural change is replacing the flat `<main>` with a step-dispatch that renders different content columns per active step.

**Imports pattern** (lines 12-24):
```typescript
import { useCallback, useEffect, useMemo, useRef, useState } from 'react';
import type { DiffModel, DiffFile, Hunk, FileReviewStatus, Thread } from '@shared/types';
import { useAppStore, actions } from './store';
import { postSessionEvent } from './api';
import { TopBar } from './components/TopBar';
// ... component imports
```

**Active step derivation — new pattern to add:**
```typescript
// Derive the active UI step from store state. Step order: summary → walkthrough → review → submission.
// activeStep is NOT stored in the server-side session; it is purely a client-side
// derived value based on which workflow stage data has arrived and what was explicitly navigated.
// Rule: if user has navigated to a step explicitly (local useState), honour that.
// Fallback: derive from state shape — summary present → summary step; walkthrough active → walkthrough; etc.
type ReviewStep = 'summary' | 'walkthrough' | 'review' | 'submission';
const [activeStep, setActiveStep] = useState<ReviewStep>('summary');
```

**Step-conditional layout pattern — new pattern:**
```typescript
// Summary step: no FileExplorer, no DiffViewer — full-width summary card + right panel (chat)
// Walkthrough/Review/Submission steps: FileExplorer (240px) + DiffViewer (1fr) + right panel (320px)
return (
  <div className="app">
    <TopBar ... />
    {/* Step nav row — two-row TopBar replaces current StageStepper */}
    <main className={cn('main', activeStep === 'summary' ? 'main--summary' : 'main--diff')}>
      {activeStep === 'summary' && <SummaryStep ... />}
      {activeStep !== 'summary' && (
        <>
          <FileExplorer ... />
          <DiffViewer ... />
          <RightPanel step={activeStep} ... />
        </>
      )}
    </main>
    <StepFooter step={activeStep} ... />
    {/* existing modals: StaleDiffModal, PendingReviewModal */}
  </div>
);
```

**Event handler pattern** (lines 140-230): All event handlers use `postSessionEvent(prKey, { type: '...', ...payload })` with `.catch(() => showToast(...))`. Preserve this pattern exactly for all new step-navigation handlers.

**cn() utility** (lines 26-28): Inline string-join helper — copy as-is to every new component file.
```typescript
function cn(...parts: Array<string | false | undefined | null>): string {
  return parts.filter(Boolean).join(' ');
}
```

## Shared Patterns

### State access
**Source:** `web/src/store.ts` — `useAppStore()` + `actions`
**Apply to:** All reworked components (SummaryStep, SubmissionPanel, TopBar)
```typescript
const state = useAppStore();
// Mutations always go through actions:
actions.setActiveStep('walkthrough');
actions.setSubmitModalOpen(false);
```

### Session event posting
**Source:** `web/src/api.ts` + `web/src/App.tsx` lines 140-180
**Apply to:** Any component that triggers server-side state changes
```typescript
// Guard: always check prKey truthy before posting (T-3-13 mitigation)
if (!prKey) return;
postSessionEvent(prKey, { type: 'walkthrough.stepAdvanced', cursor: nextCursor })
  .catch(() => showToast('Could not advance step. Retry.'));
```

### Toast error pattern
**Source:** `web/src/App.tsx` lines 101-104, 534-539
**Apply to:** StepFooter CTA buttons, any new button triggering async actions
```typescript
const showToast = useCallback((msg: string) => {
  setToast(msg);
  setTimeout(() => setToast(null), 2500);
}, []);
// Render: {toast && <div className="toast" role="status" aria-live="polite">{toast}</div>}
```

### CSS class naming convention
**Source:** `web/src/index.css` throughout
Pattern: BEM-lite — block class (`.topbar`, `.chat`, `.wsl`), element with double dash (`.topbar .primary`, `.wsl-header`), modifier with `--` suffix (`.chat--collapsed`, `.main--panel-open`). Use short, scoped class names matching existing blocks. Do not use Tailwind utility classes in component JSX except in `StaleDiffModal.tsx` (where they already appear).

### Component self-guard pattern
**Source:** `web/src/components/SubmitModal.tsx` line 72, `web/src/components/FindingsSidebar.tsx` line 35
```typescript
// Components that should only render when data is present guard at top:
if (!open || !summary) return null;
if (!selfReview) return <EmptyState />;
```
In the redesign, step components are guarded by the `activeStep` conditional in App.tsx instead, so internal guards can be removed for step-view components.
