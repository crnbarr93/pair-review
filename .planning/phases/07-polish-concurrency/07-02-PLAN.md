<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server -> browser (SSE) | AuthIdentity data (login, avatarUrl) crosses from server to browser via SSE snapshot |
| browser -> external image | Avatar img tag loads from `avatars.githubusercontent.com` — CSP must explicitly allow |
| gh CLI output -> ReviewSession | JSON from `gh api user` is parsed server-side and set on the session object |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Information Disclosure | CSP img-src | mitigate | Restrict img-src to `'self' data: https://avatars.githubusercontent.com` only — do not use a wildcard. Test assertion enforces the exact domain. |
| T-07-05 | Tampering (XSS) | TopBar.tsx | mitigate | Render login via `<span>{login}</span>` and avatar via `<img src={...}>` — never `dangerouslySetInnerHTML`. React auto-escapes text content. |
| T-07-06 | Information Disclosure | store.ts | accept | Login + avatarUrl are public GitHub profile data, not secrets. Exposing them in the browser state is by design. The token itself NEVER leaves the server process. |
| T-07-07 | Spoofing | avatar URL | accept | A malicious `gh api user` response could supply any URL. Risk is low: gh CLI is a trusted local tool, and CSP restricts img-src to `avatars.githubusercontent.com` only — other domains are blocked by the browser. |
</threat_model>
