## Component Inventory

Phase 7 adds one new UI element (auth badge in TopBar row 1) and modifies no existing component layouts.

### New: `.auth-badge` (inside `web/src/components/TopBar.tsx`)

**Placement:** TopBar row 1 — between `.spacer` and existing `.topbtn` (Settings button). Sits flush in the existing flex row.

**Normal state (identity loaded, no mismatch):**
```
[ avatar circle (20px) ] [ username text ]
```
- Avatar: `<img>` 20px × 20px, `border-radius: 50%`, `src={authenticatedUser.avatarUrl}`
- Username: `<span>` 13px, `color: var(--ink-3)`, weight 400
- Container: `display: inline-flex; align-items: center; gap: 6px;`
- No background, no border — blends with existing `.topbtn` spacing

**Mismatch state (D-03):**
```
[ warning icon (14px, var(--warn)) ] [ avatar circle (20px) ] [ username text ]
```
- Warning icon: `<Ic.warning />` at 14px, `color: var(--warn)` (`#B5791F`)
- Tooltip: `title="gh auth and GITHUB_TOKEN resolve to different users"` (native HTML tooltip per existing `.ci-pill-btn` aria-label pattern)
- Avatar remains; username remains; warning icon prepended

**Absent state (D-04, fail-open):**
- Badge element is not rendered at all — no placeholder, no spinner
- Never blocks session start

**CSS class names:**
- `.auth-badge` — flex container
- `.auth-avatar` — the `<img>` element, 20px × 20px circle
- `.auth-login` — the username `<span>`

**CSP note (from RESEARCH.md Security Domain):** The existing CSP `img-src 'self'` must be extended to include `https://avatars.githubusercontent.com` to permit the avatar `<img>` load. This is a server-side change (`server/src/http/middleware/secure-headers.ts`) with a corresponding test assertion extension in `secure-headers.test.ts`.

## Interaction Contract

### Auth badge hover

No hover state required. The badge is read-only display. It does not trigger any action on click.

### Auth badge tooltip

Use native HTML `title` attribute (consistent with existing `.ci-pill-btn` `aria-label` pattern in TopBar). No custom tooltip component needed.

### Auth badge mismatch

Warning icon appears inline to the left of the avatar. No modal, no banner, no blocking interstitial. The badge is informational only.

### No new modals, drawers, or overlays

Phase 7 introduces no new interactive overlays. All test harness work is server-side only.

## Implementation Notes for Executor

1. **Do not add new CSS classes for spacing values** — use existing inline styles matching the `.topbar` flex pattern (`gap: 6px`, `align-items: center`) consistent with how `.topbtn` and `.branch` chips are laid out.

2. **Avatar `<img>` must never use `dangerouslySetInnerHTML`** — render via standard `<img src={...} />` JSX.

3. **CSP must be patched before the avatar `<img>` is rendered** — if `secure-headers.ts` is not updated first, the avatar will be blocked by the browser's CSP and show a broken image icon. Fix the CSP as part of the same plan that adds the badge.

4. **Badge is fail-open** — the `TopBarProps` interface extension must accept `authenticatedUser?: AuthIdentity | null` with `undefined` as the default (badge not rendered). Never throw, never show a fallback spinner.

5. **Do not re-fetch identity on SSE reconnect** — identity is set once on the initial `ReviewSession` snapshot. Subsequent SSE snapshots carry it through the existing state shape without a new `gh` CLI call.
