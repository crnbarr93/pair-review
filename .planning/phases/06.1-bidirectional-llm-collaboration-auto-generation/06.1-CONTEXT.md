# Phase 06.1: Bidirectional LLM Collaboration + Auto-Generation — Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 06.1 flips the interaction model so the browser UI drives the LLM — not just the other way around. Three capabilities land here: (1) auto-generation of the PR summary and walkthrough on startup so the user never has to prompt Claude manually in the terminal, (2) a reverse communication channel so the browser can send requests to the Claude Code session via a long-poll MCP tool, and (3) two UI surfaces that use that channel — a persistent chat panel for general discussion/direction, and user-initiated inline comments with @claude tagging for line-level collaboration.

**Explicitly in scope:**
- Auto-generation of `set_pr_summary` + `set_walkthrough` on startup (not `run_self_review`).
- `--dry` flag on the slash command to skip auto-generation but keep the LLM listening.
- New MCP tool: `await_user_request` — long-poll tool that blocks until the browser sends a request. Returns the request payload; Claude processes it then re-enters the loop.
- New MCP tool: `respond_chat` — sends an LLM response message to the chat panel via SSE.
- Server-side request queue for UI→LLM requests (handles concurrency when Claude is busy processing).
- `await_user_request` timeout (~5 min) with re-enter loop to prevent indefinite blocking.
- Chat panel: right-side collapsible panel, open by default, persisted in session state via SessionEvents.
- User-initiated inline comments: gutter + icon (GitHub PR style) opens inline composer below the line.
- @claude tagging model: non-tagged comments become auto-drafted post bodies (user notes); @claude-tagged comments go through the reverse channel for LLM response.
- Self-review requestable from the UI via a "Run review" button (through the user-request queue).
- Updated slash command (`pair-review.md`) to instruct Claude to generate then enter the listen loop.
- New `SessionEvent` variants for chat messages, user-initiated threads, and request queue state.

**Explicitly NOT in scope:**
- MCP sampling (`sampling/createMessage`) — not using this; long-poll approach instead.
- Changes to existing MCP tools (the 10 tools from Phases 1–6 remain unchanged).
- Multi-session switcher, concurrency (Phase 7, SESS-04).
- Any Anthropic API call from the plugin process.

</domain>

<decisions>
## Implementation Decisions

### Auto-generation flow

- **D-01:** **Auto-generate PR summary + walkthrough on startup.** After `start_review` completes, the slash command instructs Claude to call `set_pr_summary` (generating intent/key-changes/risk-areas) and then `set_walkthrough` (building the narrative step order). Self-review (`run_self_review`) is NOT auto-generated — it's requestable on demand from the UI.
- **D-02:** **`--dry` mode skips auto-generation but keeps the LLM listening.** When the slash command receives `--dry`, it skips the summary and walkthrough generation but still enters the listen loop. The browser loads with the diff but no LLM-generated content. The user can then request generation (or anything else) from the UI.
- **D-03:** **Auto-enter the listen loop after generation completes.** The slash command flow is: `start_review` → `set_pr_summary` → `set_walkthrough` → enter `await_user_request` loop. The user never has to type anything in the terminal after invoking the slash command. With `--dry`: `start_review` → enter `await_user_request` loop.

### UI → LLM reverse channel

- **D-04:** **Long-poll MCP tool (`await_user_request`) as the reverse channel.** A new MCP tool that blocks until the browser POSTs a request to `/api/user-request`. When a request arrives, the tool resolves with the request payload (`{ type, payload }`). Claude processes the request (calling other MCP tools as needed), then calls `await_user_request` again to wait for the next request. This works within MCP's existing model — no protocol extensions needed.
- **D-05:** **Server-side request queue.** The server maintains an in-memory queue of UI requests per session. When `await_user_request` is called, it dequeues the next item (or blocks if empty). When the browser POSTs a request while Claude is busy processing, it's queued. SSE pushes a `request.queued` event to the UI so it can show a pending indicator.
- **D-06:** **Timeout with re-enter (≈5 min).** `await_user_request` times out after ~5 minutes and returns `{ type: 'no_request' }`. The slash command instructs Claude to call `await_user_request` again immediately on timeout. This prevents potential MCP transport issues from indefinite blocking while keeping the loop alive.

### Chat panel

- **D-07:** **Right-side collapsible panel, open by default.** The chat panel sits to the right of the diff viewer (≈320px). Layout becomes: FileExplorer (280px) | DiffViewer (1fr) | ChatPanel (320px). Collapsible via a toggle button. Open by default when the review workspace loads — makes the collaborative nature immediately obvious.
- **D-08:** **Chat messages are a user-request type through the reverse channel.** When the user types in the chat, the browser POSTs `{ type: 'chat', message: '...' }` to `/api/user-request`. A `chat.userMessage` SessionEvent fires immediately (so the user sees their message right away). When Claude picks it up via `await_user_request`, it processes the message and calls `respond_chat({ message })` — which fires a `chat.llmMessage` SessionEvent that SSE pushes to the browser.
- **D-09:** **Chat is persisted via SessionEvents.** Chat messages (both user and LLM) are `SessionEvent` variants through the Phase 2 reducer pipeline. History survives browser refresh and session resume. Consistent with how threads, walkthrough, and summary persist.
- **D-10:** **Chat supports full scope: PR questions, review direction, code discussion, re-generation requests.** The user can ask anything — "What's the intent of this change?", "Focus on the auth changes", "Regenerate the walkthrough", "How does this pattern work?". Claude has full context of the review session and can call any MCP tool in response.
- **D-11:** **Self-review requestable from the UI.** A "Run review" button (or similar affordance) in the UI sends `{ type: 'run_self_review' }` through the user-request queue. Claude picks it up and calls `run_self_review`. This means self-review is on-demand from the browser, not auto-generated.

### @claude inline tagging

- **D-12:** **User-initiated threads via gutter + icon.** A `+` icon appears in the diff line-number gutter on hover (GitHub PR review style). Clicking it opens an inline text composer below the line. This is the entry point for user-initiated comments.
- **D-13:** **@claude tagging model: tagged → LLM responds; untagged → user note.** Comments containing `@claude` are sent through the reverse channel for LLM processing. Claude receives the message with line context (lineId, the user's text) and responds via `reply_in_thread`. Non-tagged comments are purely user notes — they skip the LLM entirely.
- **D-14:** **Non-tagged user comments auto-create a draft body.** A non-@claude comment immediately becomes the draft post body for that thread (the text that will be posted to GitHub). No LLM synthesis needed — the user wrote exactly what they want to post.
- **D-15:** **@claude threads use `draft_comment` for synthesis.** Same as Phase 5: after the @claude conversation, Claude calls `draft_comment` to synthesize the thread into a single post body. The user can edit before submission. The LLM synthesizes, the user approves.
- **D-16:** **Phase 5 D-08 (all threads LLM-initiated) is superseded.** Phase 06.1 introduces user-initiated threads. Both LLM-initiated (from walkthrough/self-review) and user-initiated (from gutter click) threads coexist. The `Thread` type gains an `initiator` field (`'llm' | 'user'`).

### New MCP tools for Phase 06.1

- **D-17:** **Two new MCP tools: `await_user_request` + `respond_chat`.** Cumulative toolbelt: **12 tools** (Phase 1: 1, Phase 4: 4, Phase 5: 4, Phase 6: 1, Phase 06.1: 2). This exceeds the Pitfall 15 budget of ~10, but `await_user_request` is infrastructure (not a review action the LLM chooses) and `respond_chat` is a simple response channel. The LLM's active tool vocabulary for *review decisions* remains at 10.
- **D-18:** **`await_user_request` input schema:** `{}` (no params). Returns `{ type: 'chat' | 'inline_comment' | 'run_self_review' | 'regenerate_summary' | 'regenerate_walkthrough' | 'no_request', payload?: object }`. The payload varies by type.
- **D-19:** **`respond_chat` input schema:** `{ message: string }`. Fires `chat.llmMessage` SessionEvent. Returns `{ content: [{ type: 'text', text: 'Chat response sent' }] }`.

### Reducer extensions (Phase-2 event-sourcing pattern)

- **D-20:** New `SessionEvent` variants for Phase 06.1:
  - `chat.userMessage` — fires when the user sends a chat message from the browser. Stores `{ message, timestamp }`.
  - `chat.llmMessage` — fires when Claude responds via `respond_chat`. Stores `{ message, timestamp }`.
  - `thread.userStarted` — fires when a user creates a thread via the gutter + icon. Stores `{ lineId, message, isClaudeTagged }`.
  - `request.queued` — fires when a UI request is queued while Claude is busy. Stores `{ type, position }`. UI uses this for pending indicators.
  - `request.processing` — fires when Claude picks up a request from the queue. Clears the pending indicator.
  - All events follow the Phase 2 reducer/applyEvent/SSE pattern. `lastEventId` ownership remains with `applyEvent`.
- **D-21:** `ReviewSession` gains new fields (all optional for backward compat):
  - `chatMessages?: ChatMessage[]` — ordered list of chat messages (both user and LLM).
  - `requestQueue?: { pending: number }` — current queue depth for UI display.

### Claude's Discretion

The planner resolves the following without further user input:
- Exact `ChatMessage` type shape (author field, timestamp format, markdown support).
- Exact chat panel component structure (message list, input area, scroll behavior).
- Exact gutter + icon styling and hover behavior.
- Exact inline composer component (textarea, submit button, @claude detection).
- Whether `@claude` detection is prefix-only, anywhere-in-text, or a dedicated toggle/button.
- Exact timeout duration for `await_user_request` (guidance: ~5 minutes).
- Exact `request.queued` / `request.processing` indicator UI (spinner, badge, toast).
- Whether the "Run review" button lives in the TopBar, the chat panel, or both.
- Exact queue data structure and concurrency handling.
- Whether `respond_chat` supports markdown rendering in the chat panel.
- How the slash command prompt structures the generate-then-listen instruction.
- How `--dry` is parsed and propagated (slash command argument parsing, passed to `start_review`, or handled in the command prompt itself).

</decisions>

<specifics>
## Specific Ideas

- **The reverse channel is the load-bearing architectural change.** Everything else (chat, inline comments, auto-generation) is a consumer of the `await_user_request` long-poll mechanism. Get this right first — the rest follows.
- **The slash command becomes the orchestration surface.** It instructs Claude to generate, then enter an indefinite listen loop. This is a fundamental change from the current "call start_review and stop" pattern. The prompt must be clear enough that Claude follows the loop discipline without drifting.
- **@claude tagging is the collaboration boundary.** Non-tagged comments are the user's own review notes — they bypass the LLM entirely and become draft bodies immediately. @claude-tagged comments enter the pair-review model. This gives the user control over when they want LLM input and when they just want to write their own review.
- **Chat panel scope is deliberately broad.** The user can ask questions, direct the review, request re-generation — all through the same chat surface. Claude has full session context so it can respond to anything. This replaces the need to switch to the terminal to prompt Claude.
- **Phase 5 D-08 is explicitly superseded.** That decision said "every thread is LLM-initiated." Phase 06.1 introduces user-initiated threads. Both types coexist. The `Thread` type needs an `initiator` field so the UI can distinguish them (LLM threads show the walkthrough narrative; user threads show the user's comment).

</specifics>

## Deferred Ideas

- **MCP sampling (`sampling/createMessage`)** — more elegant reverse channel but unclear Claude Code support. Revisit if the long-poll approach proves unreliable.
- **Auto-run self-review on startup** — user explicitly chose not to auto-generate this. Available on demand from the UI.
- **Chat message reactions/threading** — simple linear chat is sufficient for v1 pair-review.
- **File-scoped chat context** — chat could automatically include the currently-viewed file as context. Nice to have but not essential.
- **Voice input in chat** — browser speech-to-text for the chat panel. Far future.
- **Chat export** — exporting chat history alongside the review. Not needed for v1.
