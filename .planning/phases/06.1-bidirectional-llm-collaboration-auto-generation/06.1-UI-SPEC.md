## Layout

Phase 06.1 changes the main 2-column grid to a 3-column grid when the chat panel is open.

```
TopBar       (44px, full width)
StageStepper (52px, full width)
Main grid    (1fr remaining height):
  FileExplorer (240px) | DiffViewer (1fr) | ChatPanel (320px)
```

The existing `.main--panel-open` class in index.css already declares `grid-template-columns: 240px 1fr 340px`. Phase 06.1 targets 320px for ChatPanel per D-07. Update the existing `.main--panel-open` rule:

```css
.main--panel-open {
  grid-template-columns: 240px 1fr 320px;
}
```

ChatPanel is open by default (D-07). A collapse toggle button on the chat panel header reduces the layout to the 2-column state (`.main` default: `280px 1fr`). When collapsed, FileExplorer returns to 280px.

## Component Inventory

New components introduced in Phase 06.1:

### ChatPanel (`web/src/components/ChatPanel.tsx`)

Structure:
```
.chat (flex column, border-left: 1px solid --line)
  .chat-head (44px height, avatar + name + status dot + collapse toggle)
  .chat-progress (optional: review progress mini-strip, background --paper-2)
  .chat-body (flex:1, overflow-y:auto, gap:14px)
    .msg × N (user and LLM messages)
  .chat-input (border-top, textarea + send button)
```

Reuses existing CSS classes from index.css (`.chat`, `.chat-head`, `.chat-body`, `.chat-input`, `.msg`, `.btn-sm`). These classes were pre-written in the design system as the "Chat panel (right)" section of index.css. Phase 06.1 activates this existing CSS.

ChatPanel collapse toggle: a `<button>` with `aria-label="Collapse chat"` / `aria-label="Expand chat"` in `.chat-head`. Icon: chevron-right SVG (16×16px) from icons.tsx. On click, toggles `.main` vs `.main--panel-open` on the `.main` grid container.

### InlineComposer (`web/src/components/InlineComposer.tsx`)

Gutter `+` icon:
- Position: absolute within `.diff-table .gutter`, same slot as `.thread-marker`
- Visible: on hover of a diff row that has no existing thread (`:hover` on the `<tr>`)
- Size: 16×16px, border-radius 50%, background `--paper-3` at rest, `--claude-2` on hover
- Icon: `+` character in `--claude` color on hover, `--ink-4` at rest
- Cursor: pointer
- `aria-label`: "Start comment on line {lineNumber}"

Inline composer (appears as a `thread-row` immediately below the hovered line when clicked):
```
.thread-panel (background: #FFFDF7, border-left: 3px solid --claude)
  textarea (font: inherit 13px, min-height: 60px, resize: vertical)
  .thread-reply-footer (flex row, gap: 8px)
    @claude detection chip (visible if "@claude" detected in textarea value)
    .spacer
    "Cancel" button (.btn-sm)
    "Comment" button (.btn-sm.primary — background --ink)
```

@claude detection: when the textarea contains `@claude`, show an inline chip using `--claude-2` / `--claude` tokens indicating "LLM will respond". When not present, show nothing (the comment will be a user-note only).

Composer focus style: `outline: 2px solid var(--claude); outline-offset: -1px` (inherits existing `.thread-draft-input:focus` rule).

### Request Queue Indicator

When `requestQueue.pending > 0`, show a badge in the chat header:

```
.chat-head
  ... existing content ...
  <span class="queue-badge">{pending}</span>  // new
```

Queue badge: 18×18px, border-radius 50%, background `--claude`, color `#fff`, font `--mono` 10px. Position: inline after the Claude name text. Disappears when `pending === 0`.

An optional "Thinking…" spinner: use the existing `.v-ring` CSS pattern (already defined in index.css under `.tweak-opt .preview.v-ring`). Show as a 16×16px spinning ring in `--claude` color inside the chat header when `request.processing` is active.

### "Run Review" Button

Placement: within `.chat-head` toolbar row, right-aligned (before the collapse toggle). Label: "Run review". Style: `.btn-sm` with a small checklist icon (existing icons.tsx).

When self-review is pending/running, the button shows a disabled state: `opacity: 0.5; cursor: not-allowed` with label "Running…".

## Interaction Contracts

### Chat Panel

1. User types in `.chat-input textarea` and presses Enter (without Shift) or clicks "Send".
2. Browser POSTs `{ type: 'chat', message }` to `/api/user-request`.
3. `chat.userMessage` SessionEvent fires → message appears immediately in `.chat-body` (optimistic render).
4. Queue badge increments if Claude is busy.
5. When Claude calls `respond_chat`, `chat.llmMessage` SSE event arrives → LLM message appends to `.chat-body`.
6. `.chat-body` auto-scrolls to bottom on new messages (only if user is already at or near bottom — do not forcibly scroll if user has scrolled up to read history).

Keyboard: Shift+Enter inserts a newline. Enter alone sends. This is the standard chat convention.

### Inline Composer

1. User hovers a diff row — gutter `+` icon appears.
2. User clicks `+` icon — InlineComposer mounts as a `thread-row` below the line. The hovered row gains a subtle `background: var(--paper-2)` highlight.
3. User types. If `@claude` appears in the text, the `@claude` detection chip fades in and the "Comment" button label changes to "Ask Claude".
4. User clicks "Comment" (user-note path): browser fires `thread.userStarted` event with `{ lineId, message, isClaudeTagged: false }`. Comment becomes draft body for that thread immediately.
5. User clicks "Ask Claude" (@claude path): browser fires `thread.userStarted` with `{ lineId, message, isClaudeTagged: true }` AND POSTs `{ type: 'inline_comment', lineId, message }` to `/api/user-request`. Thread card appears with the user message. Claude's response arrives via `reply_in_thread` tool call (existing Phase 5 path).
6. User clicks "Cancel" or presses Escape — InlineComposer unmounts, no event fired.
7. A line that already has an existing thread shows the `.thread-marker` (existing) instead of the `+` gutter icon.

### Auto-Generation Startup Sequence

The auto-generation itself is driven by the MCP slash command, not the browser. The browser shows:

1. At startup (before summary/walkthrough are set): DiffViewer shows with the diff, ChatPanel is open and shows the empty state with suggestion chips.
2. As `set_pr_summary` fires: SummaryDrawer fades in (existing Phase 4 behavior). A system message appears in `.chat-body`:  `"Generating summary…"` → replaced by `"Summary ready — see the Summary tab above."` once the SSE event lands.
3. As `set_walkthrough` fires: WalkthroughBanner appears (existing Phase 5 behavior). A system message in `.chat-body`: `"Walkthrough ready — {N} steps curated."`.
4. Once both are done, Claude enters the listen loop — chat status changes to "Listening…".

System messages in chat: use the existing `.msg` layout with a distinct avatar: a 22×22px neutral avatar (background `--paper-3`, border `1px solid --line`) with an italic label "System". Font color `--ink-4`. These are informational only, not part of the persisted `ChatMessage[]` history.

### Pending Indicator

When `requestQueue.pending > 0`:
- Queue badge on chat header shows the count.
- A "Thinking…" system message appears at the bottom of `.chat-body` with the spinner animation.
- When `request.processing` fires, the queue count decrements and the "Thinking…" message persists until the response arrives.

## Accessibility

- All interactive elements have `aria-label` where icon-only.
- Chat input `<textarea>` has `aria-label="Message Claude"`.
- Gutter `+` button: `aria-label="Start comment on line {N}"`.
- Chat message list: `role="log"` on `.chat-body`, `aria-live="polite"`.
- Queue badge: `aria-label="{N} messages queued"` on the badge element.
- Collapse toggle: `aria-expanded="true|false"` + `aria-controls="chat-panel"`.
- InlineComposer @claude chip: `role="status"` for detection feedback.
- Focus ring: `outline: 2px solid var(--claude)` on all interactive elements (consistent with Phase 5 `.thread-draft-input:focus`).
- `prefers-reduced-motion`: queue spinner animation wrapped in `@media (prefers-reduced-motion: reduce) { animation: none }` (consistent with existing pattern in index.css).
