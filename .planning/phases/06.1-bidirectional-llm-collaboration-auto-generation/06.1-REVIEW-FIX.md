## Fixed Issues

### WR-01: inline_comment returns `queued: false` even when isClaudeTagged and actually queued

**Files modified:** `server/src/http/routes/user-request.ts`
**Commit:** 4b488ae
**Applied fix:** When `isClaudeTagged` is true, the handler now checks `pendingCount` and fires a `request.queued` SSE event if items are already queued. The `enqueue()` call is made after the event, and the response returns `queued: queue.pendingCount > 0` to accurately reflect whether the request is waiting behind other items.

### WR-02: Chat message payload.message not validated for type or emptiness

**Files modified:** `server/src/http/routes/user-request.ts`
**Commit:** 4b488ae
**Applied fix:** Added explicit validation for the chat message payload: checks `typeof message !== 'string'` and `message.trim().length === 0` (returns 400), and enforces a max length of 8000 characters (matching the `respond_chat` MCP tool's zod schema). The validated `message` variable is used directly instead of the `as string` cast.

### WR-03: InlineComposer silently swallows postUserRequest errors

**Files modified:** `web/src/components/InlineComposer.tsx`
**Commit:** 4592354
**Applied fix:** Changed `handleSubmit` to `async`, moved `onClose()` inside the `try` block so it only fires on success. On failure, the catch block keeps the composer open so the user can see their comment is still there and retry. Removed the fire-and-forget `.catch(() => {})` pattern.

### WR-04: `darkenIfTooLight` only works correctly for 6-digit hex colors

**Files modified:** `web/src/components/DiffViewer.tsx`
**Commit:** ea97e53
**Applied fix:** Added early return guard `if (hex.length !== 7) return hex;` at the top of `darkenIfTooLight`. This ensures only `#RRGGBB` format hex strings are processed, preventing incorrect parsing of 3-digit (`#FFF`) or 8-digit (`#FFFFFFAA`) hex codes that pass the broader `HEX_COLOR` regex.

### WR-05: Request queue timeout resolve can race with a concurrent enqueue

**Files modified:** `server/src/session/request-queue.ts`
**Commit:** 3ba7989
**Applied fix:** Added identity check `if (this.resolver === resolve)` in the timeout callback before clearing state and calling `resolve(null)`. This ensures that if `enqueue()` has already consumed the resolver, the timeout callback becomes a no-op rather than double-resolving the promise.
