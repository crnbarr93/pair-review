## Summary

Phase 06.1 introduces the reverse communication channel ŌĆö the load-bearing architectural change that lets the browser drive the LLM rather than the other way around. The mechanism is `await_user_request`, a long-poll MCP tool that blocks until the HTTP layer receives a POST to `/api/user-request`, resolves the pending MCP call with the request payload, and returns control to Claude so it can process the request and call other tools. Everything else in the phase ŌĆö the chat panel, user-initiated threads, auto-generation, and the slash command rewrite ŌĆö is a consumer of this primitive.

The existing codebase provides direct analogs for every new component. The `await_user_request` tool has no existing analog in the tool files (it is fundamentally different in that it long-polls rather than returns immediately), but the Promise-based resolve pattern is standard Node.js. The HTTP route, reducer extensions, store actions, and React components all follow patterns established in Phases 3ŌĆō6. The highest-risk element is the long-poll mechanics: the Promise that blocks an MCP handler call until an HTTP POST arrives, including correct timeout, graceful re-enter, and what happens if the MCP transport drops while a Promise is pending.

The research confirms that everything needed is achievable within the existing stack with no new dependencies. The in-memory request queue can be a simple Map from `prKey` to a resolver function. Node.js is single-threaded so concurrent access safety is not a mutex problem ŌĆö it is a queue-ordering problem, which the FIFO array structure handles correctly.

**Primary recommendation:** Build `await_user_request` first (the queue module + the tool + the HTTP route), verify the round-trip in isolation, then build chat panel and inline composer on top of the working channel.

### Request Queue Module

The core new server-side module. Keeps the long-poll promise resolver in memory, paired with a FIFO queue for requests that arrive while Claude is busy:

```
server/src/session/
  request-queue.ts    # new ŌĆö SessionRequestQueue class
```

```typescript
// Conceptual shape (implementation detail for planner)
interface QueuedRequest {
  type: 'chat' | 'inline_comment' | 'run_self_review' | 'regenerate_summary' | 'regenerate_walkthrough';
  payload?: Record<string, unknown>;
}

class SessionRequestQueue {
  private queue: QueuedRequest[] = [];
  private resolver: ((req: QueuedRequest | null) => void) | null = null;
  private timeout: ReturnType<typeof setTimeout> | null = null;

  // Called by await_user_request tool handler:
  // Waits for next queued request or times out after `ms`.
  async waitForRequest(ms: number): Promise<QueuedRequest | null> { ... }

  // Called by POST /api/user-request:
  // If a waiter is pending, resolve it immediately; else push to queue.
  enqueue(req: QueuedRequest): void { ... }

  get pendingCount(): number { return this.queue.length; }
}
```

Key invariants for implementation:
- Only one resolver active at a time per session (Claude calls `await_user_request` sequentially)
- `enqueue` must clear `this.resolver` before calling it (otherwise a second `enqueue` call could double-resolve)
- On timeout, resolve with `null` (maps to `{ type: 'no_request' }` in the tool return)
- The queue module does NOT call `applyEvent` ŌĆö the HTTP route and MCP tool handlers do

### Pattern 1: Long-Poll MCP Tool Handler

The `await_user_request` tool handler is unique in the codebase: it must block until an external event arrives (HTTP POST) or a timeout fires. This is done with a Promise that the HTTP handler resolves.

```typescript
// server/src/mcp/tools/await-user-request.ts
import { z } from 'zod';
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { SessionManager } from '../../session/manager.js';
import type { RequestQueueManager } from '../../session/request-queue.js';
import { logger } from '../../logger.js';

const Input = z.object({});  // D-18: no params

const TIMEOUT_MS = 5 * 60 * 1000;  // ~5 min per D-06

export function registerAwaitUserRequest(
  mcp: McpServer,
  manager: SessionManager,
  queueManager: RequestQueueManager,
): void {
  mcp.registerTool(
    'await_user_request',
    {
      title: 'Await User Request',
      description: DESCRIPTION,
      inputSchema: Input.shape,
    },
    async (_args, extra) => {
      // The prKey is not in the schema (D-18) but must come from context.
      // Pass it via the RequestQueueManager keyed by session.
      // NOTE: MCP SDK passes a context object as the second argument;
      // the prKey must be resolved from the active session via manager.
      // Simplest approach: queueManager is keyed by prKey, and the slash
      // command always runs one session at a time (single-user tool).
      const prKey = manager.getActivePrKey();  // see Pattern 4 below
      if (!prKey) {
        return {
          content: [{ type: 'text' as const, text: 'No active session. Call start_review first.' }],
          isError: true as const,
        };
      }

      try {
        // Fire request.processing SSE to clear pending indicator
        await manager.applyEvent(prKey, { type: 'request.processing' });

        const queue = queueManager.getQueue(prKey);
        const req = await queue.waitForRequest(TIMEOUT_MS);

        if (!req) {
          // Timeout ŌĆö return no_request so slash command loops
          return {
            content: [{
              type: 'text' as const,
              text: JSON.stringify({ type: 'no_request' }),
            }],
          };
        }

        return {
          content: [{
            type: 'text' as const,
            text: JSON.stringify({ type: req.type, payload: req.payload ?? null }),
          }],
        };
      } catch (err) {
        logger.error('await_user_request failed', err);
        return {
          content: [{ type: 'text' as const, text: `await_user_request failed: ${err instanceof Error ? err.message : String(err)}` }],
          isError: true as const,
        };
      }
    }
  );
}
```

**CRITICAL: The prKey problem.** The `await_user_request` tool takes no parameters (D-18), but the queue is keyed by prKey. The SessionManager needs a way to provide the active prKey. Two options:

Option A (recommended): Add `getActivePrKey(): string | null` to `SessionManager`. Since this is a single-user tool, there is at most one active session at any given time. The `startReview` method already tracks sessions in a Map; the most recently started session is the "active" one. This is safe because the slash command always starts one review.

Option B: Accept `prKey` as an optional hidden param in the tool schema. The slash command instructs Claude to always pass it. Simpler but breaks D-18.

Option A is cleaner and matches the single-user constraint in PROJECT.md.

### Pattern 2: HTTP Route for User Requests

Follows `confirm-submit.ts` exactly:

```typescript
// server/src/http/routes/user-request.ts
import type { Hono } from 'hono';
import { z } from 'zod';
import type { SessionManager } from '../../session/manager.js';
import type { RequestQueueManager } from '../../session/request-queue.js';
import { logger } from '../../logger.js';

const bodySchema = z.object({
  prKey: z.string().min(1),
  type: z.enum(['chat', 'inline_comment', 'run_self_review', 'regenerate_summary', 'regenerate_walkthrough']),
  payload: z.record(z.unknown()).optional(),
}).strict();

export function mountUserRequest(
  app: Hono,
  manager: SessionManager,
  queueManager: RequestQueueManager,
): void {
  app.post('/api/user-request', async (c) => {
    const parsed = bodySchema.safeParse(await c.req.json().catch(() => null));
    if (!parsed.success) return c.text('Bad request', 400);

    const { prKey, type, payload } = parsed.data;
    const session = manager.get(prKey);
    if (!session) return c.text('Unknown session', 404);

    // Fire SessionEvent immediately so the UI gets feedback
    // chat.userMessage fires for chat type; thread.userStarted fires for inline_comment
    if (type === 'chat' && payload?.message) {
      await manager.applyEvent(prKey, {
        type: 'chat.userMessage',
        message: payload.message as string,
        timestamp: new Date().toISOString(),
      });
    } else if (type === 'inline_comment' && payload?.lineId && payload?.message) {
      await manager.applyEvent(prKey, {
        type: 'thread.userStarted',
        lineId: payload.lineId as string,
        message: payload.message as string,
        isClaudeTagged: true,  // only @claude comments go through this channel (D-13)
      });
    }

    const queue = queueManager.getQueue(prKey);
    const pending = queue.pendingCount;

    if (pending > 0) {
      // Claude is busy ŌĆö queue and fire request.queued SSE
      await manager.applyEvent(prKey, { type: 'request.queued', requestType: type, position: pending });
    }

    queue.enqueue({ type, payload });

    return c.json({ ok: true, queued: queue.pendingCount > 1 });
  });
}
```

Security: the `/api/*` tokenValidate middleware in `server.ts` already covers this route. No per-route middleware needed.

### Pattern 3: `respond_chat` Tool

Minimal tool ŌĆö follows `draft-comment.ts` pattern:

```typescript
// server/src/mcp/tools/respond-chat.ts
import { z } from 'zod';
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { SessionManager } from '../../session/manager.js';
import { logger } from '../../logger.js';

const Input = z.object({
  message: z.string().min(1).max(8000),
});

export function registerRespondChat(mcp: McpServer, manager: SessionManager): void {
  mcp.registerTool(
    'respond_chat',
    { title: 'Respond in Chat', description: DESCRIPTION, inputSchema: Input.shape },
    async ({ message }) => {
      try {
        const prKey = manager.getActivePrKey();
        if (!prKey) {
          return {
            content: [{ type: 'text' as const, text: 'No active session.' }],
            isError: true as const,
          };

          type: 'chat.llmMessage',
          message: message as string,
          timestamp: new Date().toISOString(),
        });
        return { content: [{ type: 'text' as const, text: 'Chat response sent.' }] };
      } catch (err) {
        logger.error('respond_chat failed', err);
        return {
          content: [{ type: 'text' as const, text: `respond_chat failed: ${err instanceof Error ? err.message : String(err)}` }],
          isError: true as const,
        };
      }
    }
  );
}
```

// In SessionManager:

// New public method:

### Pattern 5: Slash Command Rewrite (`commands/pair-review.md`)

The current slash command calls `start_review` and stops. The rewrite must:
1. Call `start_review` with the source
2. (Unless `--dry`) Call `set_pr_summary`, then `set_walkthrough`
3. Enter an indefinite loop: call `await_user_request`, process the result, repeat
4. Handle `{ type: 'no_request' }` by immediately looping (re-call `await_user_request`)

The `allowed-tools` frontmatter restricts which tools Claude can call. Phase 06.1 must expand this to include all 12 tools:

```markdown
---
description: Open a PR review workspace in the browser with bidirectional collaboration
argument-hint: <pr-url-or-number> | --local <base> <head> [--dry]
allowed-tools: mcp__git-review-plugin__start_review, mcp__git-review-plugin__set_pr_summary, mcp__git-review-plugin__set_walkthrough, mcp__git-review-plugin__run_self_review, mcp__git-review-plugin__list_files, mcp__git-review-plugin__get_hunk, mcp__git-review-plugin__reply_in_thread, mcp__git-review-plugin__draft_comment, mcp__git-review-plugin__resolve_thread, mcp__git-review-plugin__submit_review, mcp__git-review-plugin__await_user_request, mcp__git-review-plugin__respond_chat
---
```

The command body instructs Claude with an explicit loop protocol:

```
STARTUP SEQUENCE (unless --dry flag is present):
1. Call start_review with the source from user input.
2. Call set_pr_summary to generate the PR summary.
3. Call set_walkthrough to generate the walkthrough narrative.

LISTEN LOOP (enter after startup, or immediately if --dry):
4. Call await_user_request. This blocks until the browser sends a request or times out.
5. Process the returned request based on its type:
   - type: 'no_request' ŌåÆ immediately call await_user_request again (loop)
   - type: 'chat' ŌåÆ call respond_chat with your answer to payload.message
   - type: 'inline_comment' ŌåÆ call reply_in_thread with the lineId and message from payload, then call draft_comment when ready
   - type: 'run_self_review' ŌåÆ call run_self_review, then call await_user_request
   - type: 'regenerate_summary' ŌåÆ call set_pr_summary, then call await_user_request
   - type: 'regenerate_walkthrough' ŌåÆ call set_walkthrough, then call await_user_request
6. After processing any request type (except no_request, which already loops), call await_user_request again.

LOOP DISCIPLINE: You MUST always re-enter the loop by calling await_user_request after processing any request. Do not exit the loop or stop calling await_user_request until the user explicitly closes the review.
```

### Pattern 6: Reducer Extensions

Add 5 new cases to `server/src/session/reducer.ts` before the `default:` exhaustiveness guard. Follow the exact spread-and-patch pattern used for all prior phases:

```typescript
// In applyEvent switch (before default:):
case 'chat.userMessage':
  return {
    ...s,
    chatMessages: [
      ...(s.chatMessages ?? []),
      { author: 'user' as const, message: e.message, timestamp: e.timestamp },
    ],
  };
case 'chat.llmMessage':
  return {
    ...s,
    chatMessages: [
      ...(s.chatMessages ?? []),
      { author: 'llm' as const, message: e.message, timestamp: e.timestamp },
    ],
  };
case 'thread.userStarted': {
  const newThreadId = `th_${nanoid(10)}`;
  const now = e.timestamp ?? new Date().toISOString();
  const newThread: Thread = {
    threadId: newThreadId,
    lineId: e.lineId,
    path: e.path ?? '',       // resolved from lineId by HTTP route before firing event
    line: e.line ?? 0,
    side: e.side ?? 'RIGHT',
    preExisting: false,
    initiator: 'user' as const,
    turns: [{ author: 'user' as const, message: e.message, createdAt: now }],
    resolved: false,
    createdAt: now,
  };
  return {
    ...s,
    threads: { ...(s.threads ?? {}), [newThreadId]: newThread },
  };
}
case 'request.queued':
  return {
    ...s,
    requestQueue: { pending: (s.requestQueue?.pending ?? 0) + 1 },
  };
case 'request.processing':
  return {
    ...s,
    requestQueue: { pending: Math.max(0, (s.requestQueue?.pending ?? 0) - 1) },
  };
```

**IMPORTANT:** The reducer is a pure function (no I/O, no nanoid). But `thread.userStarted` must create a new threadId ŌĆö which requires nanoid. Two options:

Option A (recommended): The HTTP route generates the threadId before firing the event, and the event carries it: `{ type: 'thread.userStarted', ..., threadId: 'th_xyz' }`. The reducer just stores it. This keeps the reducer pure.

Option B: The reducer calls nanoid. This violates the "pure function" invariant stated in the reducer's JSDoc comment (line 7 of reducer.ts). Do NOT use this option.

Use Option A ŌĆö the HTTP route `/api/user-request` generates the threadId using `nanoid`, resolves the lineId to `(path, line, side)`, and embeds all resolved fields in the event. The reducer just spreads.

### Anti-Patterns to Avoid

- **Resolving lineId inside the reducer:** The reducer must remain pure (no diff lookups). The HTTP route resolves lineId ŌåÆ `(path, line, side)` before firing the event, same as the MCP tools do.
- **Using `console.log` in any server file:** Corrupts the JSON-RPC stdio channel. Use `logger.error` / `logger.warn` from `../../logger.js`.
- **Double-resolve on the queue Promise:** The `enqueue` method must clear `this.resolver = null` before calling `resolve(req)`. If it does not, a race between timeout and enqueue could call the resolver twice (though in practice Node's event loop makes this rare, it is not guaranteed).
- **Forgetting to add new events to `session-events.ts` discriminated union:** Only user-browser-triggered events belong there. The new Phase 06.1 events (`chat.userMessage`, `chat.llmMessage`, `thread.userStarted`, `request.queued`, `request.processing`) are all server-fired events (via MCP tool or HTTP route) ŌĆö do NOT add them to `session-events.ts`. They go through the HTTP POST to `/api/user-request` or MCP tool call, not through the existing `/api/session/events` route.
- **Adding `prKey` to `await_user_request` schema:** D-18 says no params. Use `getActivePrKey()` instead.
- **Storing the pending resolver in the reducer state:** The queue is in-memory and ephemeral; it must NOT be persisted to SQLite or stored in `ReviewSession`. It lives in `RequestQueueManager` only.

## Research Area Findings

### Area 1: Long-Poll MCP Tool Pattern

**Verified pattern:** [VERIFIED: codebase analysis]

MCP tool handlers are async functions. The MCP SDK (`@modelcontextprotocol/sdk/server/mcp.js`) invokes them and awaits their return. This means a handler can `await` any Promise ŌĆö including one that does not resolve until an external event fires. This is the long-poll model:

```
Claude calls await_user_request
  ŌåÆ MCP SDK invokes async handler
  ŌåÆ handler calls queue.waitForRequest(timeout)
  ŌåÆ waitForRequest returns a Promise
  ŌåÆ MCP SDK is awaiting that Promise ŌĆö Claude's tool call is "in flight"
  ŌåÆ When browser POSTs to /api/user-request:
    ŌåÆ HTTP handler calls queue.enqueue(req)
    ŌåÆ enqueue calls this.resolver(req) ŌĆö resolves the Promise
    ŌåÆ waitForRequest returns the request
    ŌåÆ MCP handler returns { content: [...] }
    ŌåÆ MCP SDK delivers the result to Claude
    ŌåÆ Claude's tool call completes
```

The key insight: the MCP SDK's stdio transport is not blocking the OS thread while awaiting the Promise. Node.js event loop continues processing HTTP requests from Hono during the await. This is why the HTTP server and MCP server can share the same Node process.

**Timeout mechanics:** Use `setTimeout` to set a fallback resolver:

```typescript
async waitForRequest(ms: number): Promise<QueuedRequest | null> {
  // If queue already has items, return immediately
  if (this.queue.length > 0) {
    return this.queue.shift()!;
  }

  return new Promise<QueuedRequest | null>((resolve) => {
    this.resolver = resolve;
    this.timeout = setTimeout(() => {
      this.resolver = null;
      resolve(null);  // null ŌåÆ tool returns { type: 'no_request' }
    }, ms);
  });
}
```

**Re-enter safety:** After `waitForRequest` returns (whether via enqueue or timeout), `this.resolver` is `null` and `this.timeout` has fired or been cleared. The next `await_user_request` call starts fresh.

**Concurrent call guard:** If Claude somehow calls `await_user_request` twice before the first resolves (e.g., slash command bug), the second call would overwrite `this.resolver`. Add a guard:

```typescript
if (this.resolver !== null) {
  throw new Error('await_user_request already pending for this session');
}
```

[ASSUMED: The MCP SDK guarantees sequential tool invocations per session ŌĆö a second call would not arrive while the first is still awaiting. But the guard is cheap insurance.]

### Area 2: Server-Side Request Queue

**Design:** [VERIFIED: codebase analysis ŌĆö existing patterns]

```typescript
// server/src/session/request-queue.ts

export class SessionRequestQueue {
  private queue: QueuedRequest[] = [];
  private resolver: ((req: QueuedRequest | null) => void) | null = null;
  private timeout: ReturnType<typeof setTimeout> | null = null;

  enqueue(req: QueuedRequest): void {
    if (this.resolver !== null) {
      // Claude is waiting ŌĆö deliver immediately
      const resolve = this.resolver;
      this.resolver = null;
      if (this.timeout !== null) {
        clearTimeout(this.timeout);
        this.timeout = null;
      }
      resolve(req);
    } else {
      // Claude is busy ŌĆö queue for next await_user_request call
      this.queue.push(req);
    }
  }

  async waitForRequest(ms: number): Promise<QueuedRequest | null> {
    if (this.queue.length > 0) return this.queue.shift()!;
    if (this.resolver !== null) throw new Error('already waiting');
    return new Promise((resolve) => {
      this.resolver = resolve;
      this.timeout = setTimeout(() => {
        this.resolver = null;
        this.timeout = null;
        resolve(null);
      }, ms);
    });
  }

  get pendingCount(): number { return this.queue.length; }
}

// Manager: one queue per session
export class RequestQueueManager {
  private queues = new Map<string, SessionRequestQueue>();

  getQueue(prKey: string): SessionRequestQueue {
    if (!this.queues.has(prKey)) {
      this.queues.set(prKey, new SessionRequestQueue());
    }
    return this.queues.get(prKey)!;
  }
}
```

**Concurrency safety in single-threaded Node:** Node.js processes one callback at a time. The `enqueue` and `waitForRequest` methods cannot interleave mid-execution because neither contains `await`. The `enqueue` call from the Hono HTTP handler and the `waitForRequest` return from the MCP handler are sequential, not concurrent. No mutex needed. [VERIFIED: Node.js event loop model ŌĆö single-threaded, callback-based concurrency]

**Where `RequestQueueManager` is instantiated:** In `server/src/index.ts`, alongside `SessionManager`. Both are passed to `registerAllTools` and `buildHttpApp`. This requires adding `RequestQueueManager` to both function signatures.

**SSE `request.queued` accuracy:** The HTTP route fires `request.queued` only when `queue.pendingCount > 0` BEFORE enqueuing. So: if Claude is waiting (resolver !== null), enqueue delivers immediately without firing `request.queued`. If Claude is busy (resolver === null), `pendingCount > 0` fires the SSE. This is the correct semantic.

### Area 7: Integration Risks

**Risk 1: Claude exits the listen loop**

When Claude judges a task "complete," it tends to stop calling tools unless instructed otherwise. The slash command must be explicit: "You MUST continue calling `await_user_request` indefinitely. The loop only ends when the user closes the browser tab."

Mitigation: The `no_request` return value (on timeout) is the signal that Claude should re-enter. If Claude returns `{ type: 'no_request' }` and stops, the chat status changes to "Not connected." This is visible to the user and they can re-invoke the slash command. Not ideal but recoverable. [ASSUMED: Claude Code session behavior ŌĆö no formal guarantee on indefinite tool loops]

**Risk 2: MCP transport timeout on long await**

The MCP stdio transport does not have a documented timeout for individual tool calls. `await_user_request` could block for 5 minutes. The MCP SDK's `StdioServerTransport` uses Node.js stdio which does not inherently timeout. Claude Code may have its own timeout for tool calls. [ASSUMED: no official timeout documented; 5 minutes is shorter than most HTTP session timeouts and should be safe]

Mitigation per D-06: The 5-minute timeout returns `{ type: 'no_request' }` before any hypothetical transport timeout. This keeps the tool call short enough.

**Risk 3: Browser refresh mid-conversation**

When the browser refreshes, the SSE reconnects. The `/api/events` handler sends a full snapshot immediately (existing behavior). The `chatMessages` array is persisted in the SQLite session state via the reducer ŌåÆ `writeState` pipeline. So chat history survives refresh automatically.

However: the `requestQueue.pending` count in `ReviewSession` is a snapshot of the in-memory queue state. After refresh, the SSE snapshot shows whatever `pending` count was last persisted. If requests were queued but Claude has not yet processed them, the pending count may be stale after refresh. This is acceptable for v1 ŌĆö the chat will eventually show Claude's response when he processes the queued request.

**Risk 4: Claude context overflow in long sessions**

If the review session involves many chat messages and many tool calls, Claude's context window may fill. When the context is near full, Claude Code typically shows a warning and the user can start a new session. The listen loop would end when Claude can no longer make tool calls.

Mitigation: no code-level mitigation needed in Phase 06.1. This is a fundamental constraint of the LLM-driver model documented in PROJECT.md.

**Risk 5: `await_user_request` called before `start_review`**

If the slash command is invoked incorrectly (e.g., in the wrong order), `getActivePrKey()` returns `null`. The tool handler returns an `isError` response. The session continues working normally.

**Risk 6: Queue grows unboundedly if Claude crashes**

If the MCP process crashes while requests are queued, the in-memory queue is lost. The browser shows requests as "queued" indefinitely (since `requestQueue.pending` in the persisted session state still shows > 0). 

Mitigation: On `startReview` (session resume), reset `requestQueue.pending` to 0 in the reducer. The session resume path already runs `applyEvent` for stale-diff detection; add a `request.processing` event (which decrements pending) or just reset the field in the resume logic. Alternative: persist `requestQueue` as `{ pending: 0 }` always on disk, and only track the real count in memory. The simplest fix: on `startReview` with a persisted session, reset `requestQueue` to `{ pending: 0 }` before returning.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | ŌĆö |
| V3 Session Management | yes | Existing `tokenValidate` middleware covers `/api/user-request` automatically |
| V4 Access Control | yes | `tokenValidate` on all `/api/*` routes; no additional control needed |
| V5 Input Validation | yes | Zod `bodySchema` on `/api/user-request`; `message` field max-length enforced |
| V6 Cryptography | no | ŌĆö |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Prompt injection via chat message | Tampering | Chat `message` rendered as React text node, never innerHTML. Server stores raw text via SessionEvent. No injection into Claude's system prompt. |
| XSS via LLM-authored chat responses | Tampering | `chat.llmMessage.message` rendered as React text node in ChatPanel (same pattern as `ThreadCard.tsx` turn.message). NEVER `dangerouslySetInnerHTML`. |
| CSRF on `/api/user-request` | Tampering | Double-submit token (X-Review-Token header + `review_session` cookie) inherited from Phase 1 pattern; automatically enforced by `tokenValidate`. |
| Queue flooding (sending many requests rapidly) | Denial of Service | Server-side: queue is bounded by available memory (single user, not a concern). Could add max queue depth (e.g., 10 items) in `SessionRequestQueue` for safety. |
| lineId forgery in inline comment | Tampering | HTTP route calls `resolveLineIdExtended(session.diff, lineId)` before firing event. Invalid lineIds return 400. Existing pattern from `reply-in-thread.ts`. |
| `@claude` tag stripping/injection | Spoofing | Detection is client-side only (UI chip). Server side: `isClaudeTagged` is a boolean the client sends; server does not re-verify the message text. Acceptable for single-user tool ŌĆö no multi-user trust model. |
